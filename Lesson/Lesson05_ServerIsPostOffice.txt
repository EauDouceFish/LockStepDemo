================================================================================
LESSON 05 — 服务器是邮局不是裁判（概念课）
完成日期：2026-05-10
================================================================================

【这节课做了啥】
纯概念课，无代码。建立"lockstep 服务器 = 邮局"的心智模型，
理解为什么服务器不跑游戏逻辑。

【一句话】
纯 lockstep 服务器只干两件事：收输入、广播帧。
不跑游戏逻辑、不维护 World、不知道谁打谁。

【两种网络模型对比】

邮局模型（lockstep）：
- 火影手游 / 王者 / DOTA / SC 等 RTS、格斗
- 服务器收 N 客户端的"本帧输入信封"，凑齐就广播
- 不拆信、不读内容、不算结果

裁判模型（state sync）：
- FPS（CS / Valorant）、MMO
- 服务器自己跑完整游戏，客户端发"我想做什么"
- 服务器算结果，下发状态

我们做帧同步 = 邮局。

【邮局模型的核心收益】

维度对比（邮局 vs 裁判）：
- 服务器 CPU：几乎 0 vs 完整游戏逻辑
- 服务器内存：缓存几帧输入 KB 级 vs 完整 World MB 级
- 带宽下行：30Hz × 4玩家 × 3B ≈ 360B/s vs 每帧 World diff KB 级
- 运维成本：单机扛上千场 vs 几十场满载
- 代码规模：服务器 1-2 文件 vs 复制一份完整游戏逻辑

火影手游、王者荣耀都是这个模型。
腾讯能让 PVP 房间几十万并发，全靠这个 CPU 省。

【服务器具体职责（Lesson 7-8 才实现）】

1. 接 JoinRoom：玩家进房，分配 playerId
2. 房间满 → 广播 RoomReady（含房间种子 + 玩家列表）
3. 进入 Running，30Hz Tick：
   - 每帧从所有玩家收集 FrameInput
   - 凑齐 OR 超时（200ms）→ 打包 FrameAuth 广播
   - 重复
4. 收到 GameOver：清理房间

服务器不做的事：
- 移动计算
- 命中判定
- 扣血
- 状态机推进
- 随机数生成（客户端各自用同种子 FRandom 自己算）

【思考点：防作弊怎么办？】

A：v1 不管，v3 才管。

为什么单机改本地客户端几乎无效：
- 玩家 A 改本地"被打中→无敌"
- A 自己看到无敌，但他广播的还是"我被打中"
- 其他客户端按规则推帧 → A 在别人 World 里照常掉血倒地
- A 改的只是表现层，逻辑层在所有客户端是一致的

真正能作弊的攻击面：
1. 加速器——改本地时钟提前出招（服务器看帧号异常踢）
2. 透视/锁血——客户端看到全部信息（lockstep 死穴，因为它要跑全 World）
3. 断线重连卡帧（v2 处理）

防御策略（v3 才做）：
- 客户端上报 World hash，服务器对账分叉 kick（Lesson 33）
- 关键事件服务器复算 sanity check
- 帧速/心跳异常踢人

速通 v1 信任客户端，根本不开发这条线。

【铁律提醒】
- 服务器代码必须能在不引用 UnityEngine 的情况下编译
- 这意味着 Framework/Core / Framework/Input / Framework/Network 都不能依赖 UnityEngine
- Server/ 目录下的代码也是纯 .NET 进程
