## 2. 总结结论

当前代码不是“不可救的堆砌”，已有三项方向正确的设计：

1. 逻辑层消费文本和静态数据、不直接依赖 Unity 与音频设备，符合锁步模拟边界。
2. SFF/PCX/SND 二进制读取器独立于 Unity，具备可单测性。
3. 动画运行态保存在 `MChar`，静态动画数据共享，符合快照和回滚的基本要求。

但“导入任意 MUGEN 角色并完整适配”目前不成立。主要阻断项如下：

| 严重度 | 缺口 | 直接后果 |
|---|---|---|
| P0 | `localcoord` 只解析、几乎不参与运行 | 非 320 坐标角色的速度、位置、常量、碰撞框、动画偏移和自定义状态均产生系统性误差 |
| P0 | `ChangeAnim2` 通过永久替换 `MChar.AnimTable` 表达所有权 | 自定义状态结束后仍可能读取对方动画；`ChangeAnim`、`animexist`、`selfanimexist` 语义污染 |
| P0 | 没有独立的 `anim owner` 与 `sprite owner` | 无法准确表达 Ikemen 的 `animPN`/`spritePN`，跨角色动画、精灵、调色板和碰撞缩放不能闭环 |
| P0 | SFFv2 PNG 10/11/12 不支持 | 大量 MUGEN 1.1 角色直接缺图；当前表现层明确跳过这些精灵 |
| P0 | ACT/SND 只可解析，未进入生产角色包和表现消费链 | 调色板选择不生效；`PlaySnd` 事件无人消费，角色实际无声 |
| P1 | SFFv1 调色板被简化为“全文件找第一张非空调色板” | `same palette`、独立调色板、外部 ACT 与 palette remap 行为错误 |
| P1 | AIR 解析不支持 `Copy Action`、插值、完整帧参数，重复 action 后者覆盖前者 | 动画帧、碰撞框、混合和动作复制与 Ikemen 不一致 |
| P1 | 没有生产级角色包加载器和资源注册表 | 测试能拼装文本，但 Unity 运行路径仍硬编码文件名，不能可靠加载任意 DEF |
| P1 | DEF 不支持语言覆盖、Shift-JIS 路径、Ikemen 搜索目录/压缩包语义 | 日文角色、大小写差异包、共享 `data/` 资源和压缩资源加载失败 |
| P2 | 二进制读取器错误策略与诊断不统一 | 坏包可能被静默截断、返回空图，难以给出可操作兼容报告 |

架构判断：

- 将文件 IO 留在适配层、将声音输出为帧事件，是**合理适配差异**。
- 将动画与 SFF 解耦为逻辑数据和表现数据，也是**合理适配差异**，前提是两者由同一个不可变资源包 ID 关联。
- 用一个可变 `AnimTable` 引用同时承担“自己的动画表”“当前动画表”“状态所有者动画表”，属于**不规范简化**，会造成行为差异。
- 再增加 `SelfAnimTable`、`StateAnimTable` 等更多字典字段，只会形成“双表/多表补丁”。正确方向是资源注册表和所有者 ID。

## 3. Ikemen 的目标语义模型

Ikemen 没有“动画双表”这一抽象。它把以下概念分开：

- `CharGlobalInfo.animTable`：某个玩家槽位拥有的静态动画表。
- `CharGlobalInfo.sff` / `palettedata` / `snd`：同一玩家槽位拥有的精灵、调色板和声音资源。
- `StateBytecode.playerNo`：当前状态代码的所有者。
- `Char.playerNo`：角色自身资源所有者。
- `Char.animPN`：当前动画定义来自哪个玩家。
- `Char.spritePN`：当前动画帧应从哪个玩家的 SFF/调色板取精灵。
- `Char.localscl`：当前状态解释使用的坐标比例。
- `Char.animlocalscl`：当前动画所有者的坐标比例。

`ChangeAnim2` 的默认语义是：

1. 动画表来自状态所有者 `crun.ss.sb.playerNo`。
2. 精灵仍来自被控制角色自己的 `c.playerNo`。
3. 成功切换后记录 `animPN` 与 `spritePN`，而不是替换任意角色的静态动画表引用。
4. 碰撞缩放随动画所有者更新，精灵缩放随精灵所有者与自身 localcoord 比例更新。

因此，C# 需要的是“资源所有权上下文”，不是两份动画表。

## 4. DEF 与角色包逐函数审计

### 4.1 `MDefParser.ParseFiles`

- **Ikemen 语义**：角色加载直接解析 DEF 的 `[Info]`、`[Files]` 及语言覆盖段，并在后续通过 `LoadFile` 解析搜索路径。
- **当前 C# 语义**：调用 `Parse(text).Files`，仅返回未解析路径。
- **合理适配差异**：纯解析器不执行 IO，有利于测试和锁步层隔离。
- **行为差异/冗余**：调用者若只用 `ParseFiles` 会丢失 `localcoord`、`pal.defaults` 等解释资源所需元数据；测试加载器用它猜主 DEF，生产路径没有等价包加载器。
- **建议重构**：保留该便捷函数，但主入口必须返回完整 `MCharacterPackageManifest`，禁止生产代码只依据 `MDefFiles` 拼装角色。
- **缺测试**：多个 DEF 候选、缺 `cmd/cns` 但合法的 Ikemen 角色、语言覆盖后的主文件判定。

### 4.2 `MDefParser.Parse`

- **Ikemen 语义**：`Char.load` 与选择界面加载均支持语言前缀段；文本读取可处理 Shift-JIS；只处理相应首个基础段和首个语言段；后续资源解析使用 DEF 路径、空根目录和 `data/` 搜索目录。
- **当前 C# 语义**：逐行解析所有 `[info]`、`[files]`，后出现的值覆盖先出现的值；不感知语言、编码、来源路径和搜索根。
- **合理适配差异**：解析器接收已经解码的字符串本身合理。
- **行为差异/冗余**：编码和语言信息在进入解析器前已经丢失；重复 section 的覆盖顺序与 Ikemen 不同；结果没有 DEF 自身路径，无法形成可复现的资源定位上下文。
- **建议重构**：输入改为 `MDefSource { Text, DefPath, Encoding, Language }` 或由外层包加载器提供这些上下文；解析结果保留诊断和原始键来源。
- **缺测试**：`en.Info`/`ja.Files`、重复 section、BOM/Shift-JIS、混合大小写、CR/LF、空文本和非法 section。

### 4.3 `MDefParser.AssignInfo`

- **Ikemen 语义**：读取名称、显示名、作者、lifebar 名、`localcoord`、`portraitscale` 等；角色运行时立即据 `localcoord` 初始化比例。`pal.defaults` 主要用于选择界面调色板默认逻辑。
- **当前 C# 语义**：只保留 `name`、`displayname`、`localcoord`、`pal.defaults`。
- **合理适配差异**：本模块可以暂不承担纯 UI 字段。
- **行为差异/冗余**：`localcoord` 被存储但没有传播到运行角色；高度仅保存，后续没有宽高或宽高比策略；非法单值 localcoord 会保留默认高度，缺诊断。
- **建议重构**：将坐标定义改为不可变 `MCharacterCoordinateSpace`，加载时验证宽高并计算标准化比例，但不要在导入期改写所有数值。
- **缺测试**：`localcoord=640,480` 的运行效果、单值/零/负数/溢出、非 4:3、`portraitscale`。

### 4.4 `MDefParser.AssignFile`

- **Ikemen 语义**：读取 CNS/SFF/AIR/SND、多个状态文件、最多配置上限内的 ACT、字体等；路径经 Shift-JIS 解码和多根目录搜索；语言段可以覆盖基础段。
- **当前 C# 语义**：保存 `cmd/cns/st*/stcommon/sprite/anim/sound/palN` 原始字符串。
- **合理适配差异**：逻辑层不解析实际路径是正确边界。
- **行为差异/冗余**：没有路径来源、规范化结果、缺失资源诊断和最大 palette 槽限制；`st` 顺序仅依赖文本出现顺序；生产代码没有统一消费者。
- **建议重构**：在 IO 适配层增加 `IMugenResourceResolver`，输出已解析且保留原始路径的资源句柄；所有 View 和测试共用同一解析器。
- **缺测试**：大小写不敏感路径、反斜杠/斜杠、父目录、`data/` 回退、缺失可选/必需资源、重复 `stN`、超过 palette 上限。

### 4.5 `ParseInts` / `IsDigits` / `StripComment` / `Unquote`

- **Ikemen 语义**：INI 工具统一处理 CSV、文本、注释和引号。
- **当前 C# 语义**：宽松忽略不能解析的整数；分号后一律截断；只移除一对双引号。
- **合理适配差异**：小型纯函数足够处理常见 DEF。
- **行为差异/冗余**：引号内分号会被误当注释；坏 CSV 被静默部分接受；无转义和来源位置，兼容报告无法定位。
- **建议重构**：复用一个带行号的 INI tokenizer，不要让 DEF/AIR/CNS 各自实现不同的注释和引号规则。
- **缺测试**：带分号路径、尾随注释、空 CSV 项、十六进制/符号值、未闭合引号。

### 4.6 `MCharLoader.Load`

- **Ikemen 语义**：`Char.load` 以 DEF 为根统一加载状态、常量、SFF、独立 palette 数据、AIR、common AIR、SND，并把这些资源绑定到同一 `CharGlobalInfo`。
- **当前 C# 语义**：只消费调用者传入的 CNS/common/AIR/CMD 文本，创建 `MCharData`；DEF、SFF、ACT、SND 仅作为旁路元数据存在。
- **合理适配差异**：逻辑装配器不做文件 IO 是正确的。
- **行为差异/冗余**：`MCharData` 不是完整角色包，资源完整性只能由调用者自行保证；Unity View 与测试各自拼路径和文本，形成重复、易漂移的装配代码。
- **建议重构**：拆成 `MCharacterPackageImporter`（IO/编码/资源）和 `MCharCompiler`（纯文本到逻辑数据）。最终产物包含稳定 `PackageId` 及逻辑、视觉、声音资源清单。
- **缺测试**：从单个 DEF 端到端加载、缺 SFF/AIR/SND 的策略、common AIR 合并、重复动作优先级、资源诊断快照。

### 4.7 `MCharLoader.SpawnChar` / `ApplyInitialStateHeader`

- **Ikemen 语义**：运行角色保存 `playerNo`，通过全局 `cgi[playerNo]` 访问静态资源；初始化 anim/sprite owner 和 localcoord 比例。
- **当前 C# 语义**：把 `data.Anims` 直接赋给可变 `MChar.AnimTable`，未设置资源包 ID、localcoord、palette bank 或 sound bank。
- **合理适配差异**：静态动画字典共享且不进入快照是合理的。
- **行为差异/冗余**：共享字典引用同时被当作当前动画所有权，之后又会被 `ChangeAnim2` 改写；角色实例无法稳定回到“自己的资源”。
- **建议重构**：角色只保存 `SelfResourceOwnerId`、`AnimOwnerId`、`SpriteOwnerId`、`StateOwnerId`；静态表只能经注册表按 ID 查询。
- **缺测试**：生成两个不同包角色、helper 继承包、clone/rollback 后 owner ID 保持、初始非 320 localcoord。

## 5. AIR 与动画静态资源逐函数审计

### 5.1 `AirParser.ParseFile`

- **Ikemen 语义**：`LoadText` 支持引擎文件系统、压缩包和编码，然后由统一动画读取器解析。
- **当前 C# 语义**：直接 `File.ReadAllText`。
- **合理适配差异**：作为低层工具函数可以保留。
- **行为差异/冗余**：默认 UTF-8/系统检测不能覆盖常见老角色编码，也绕过统一资源解析器。
- **建议重构**：生产代码不调用该函数；仅由 `IMugenResourceResolver.ReadText` 提供文本给 `Parse`。
- **缺测试**：Shift-JIS AIR、压缩包 AIR、大小写路径。

### 5.2 `AirParser.Parse`

- **Ikemen 语义**：`ReadAnimation` 支持默认/单帧 Clsn、`LoopStart`、`Copy Action`、空 action 复制后续 action、插值 offset/scale/angle/blend、完整帧属性和总时长规则。
- **当前 C# 语义**：支持基础 action、Clsn1/2、LoopStart、group/image/offset/duration/flip。
- **合理适配差异**：逻辑层先保留命中所需最小字段可以作为阶段性实现。
- **行为差异/冗余**：`Copy Action`、空 action 复制、插值、alpha/blend、scale/rotation 等全部缺失；任意 `[` 行都创建 action，非法 section 可能成为 action 0；坏行直接抛异常且无行号诊断。
- **建议重构**：不要继续在当前循环里堆条件。建立 AIR tokenizer、`MAnimationDefinition` 完整中间模型，再分别编译逻辑帧和表现帧。
- **缺测试**：Copy Action 链和环、空 action、插值、负一永久帧、中间 `-1`、零时长、完整 flip/blend、错误行号。

### 5.3 `AirParser.ToDictionary`

- **Ikemen 语义**：重复 action number 使用第一个，后续项忽略并记录警告。
- **当前 C# 语义**：`map[id] = anim`，最后一个覆盖第一个。
- **行为差异**：明确不兼容，会运行不同动作。
- **建议重构**：统一在动画表构建器执行 first-wins，并向兼容报告写入重复项。
- **缺测试**：同文件重复 action、主 AIR 与 common AIR 重复时的优先级。

### 5.4 `FlushAction`

- **Ikemen 语义**：空 action 可能复制下一个 action；最终还解析 `Copy Action`。
- **当前 C# 语义**：无条件把空帧 action 加入结果。
- **行为差异**：空动画的含义不同。
- **建议重构**：先存未解析引用，所有 AIR 合并完成后做一次 copy-action 解析，和 Ikemen `resolveCopyAction` 的时机一致。
- **缺测试**：跨 common AIR 的复制目标、最多 8 层链和循环。

### 5.5 `ParseActionId` / `ParseBox` / `ParseFrame` / `ParseInt`

- **Ikemen 语义**：规范化碰撞框坐标顺序；帧解析对可选参数有完整规则；解析失败通常记录兼容警告而不是无上下文崩溃。
- **当前 C# 语义**：直接按固定下标和 `int.Parse` 读取；Clsn 坐标未排序；帧只取第六项 flip。
- **行为差异**：反向 Clsn 坐标可能保持错误方向；短帧、扩展参数、表达异常会终止整个角色导入。
- **建议重构**：返回 `TryParse` 结果和结构化诊断，规范化 Clsn，完整保留帧扩展字段。
- **缺测试**：反向 Clsn、短帧、额外逗号、非法整数、负 group/image、扩展透明参数。

### 5.6 `MAnimImport.FromAir` / `FromAirFrame` / `ConvertClsn`

- **Ikemen 语义**：动画对象持有 SFF/Palette 上下文，帧保留所有绘制与碰撞信息。
- **当前 C# 语义**：将通用 `AnimData` 复制为固定点 `MAnimData`，预计算节拍，Clsn 转成逻辑类型。
- **合理适配差异**：导入模型与运行模型分离、固定点转换、只把碰撞数据放入逻辑层，都是合理设计。
- **行为差异/冗余**：上游 `AnimData` 已丢失的绘制字段无法恢复；动画本身没有静态 `ResourceOwnerId` 或 package 关联。
- **建议重构**：逻辑动画保持无 Unity 依赖，但应保存不可变 `AnimationResourceKey { PackageId, ActionNo }` 或由解析上下文提供 owner；表现模型单独保存完整帧绘制属性。
- **缺测试**：逻辑/表现双产物一致性、Clsn 与 localcoord 缩放、空帧和永久帧节拍。

### 5.7 `MAnimImport.FromAirTable`

- **Ikemen 语义**：重复 action first-wins，全部来源合并后解析 Copy Action。
- **当前 C# 语义**：最后一个覆盖第一个。
- **行为差异**：与 `AirParser.ToDictionary` 一样明确不兼容。
- **建议重构**：删除重复的表构建策略，改由唯一 `MAnimationTableBuilder` 负责优先级、复制和诊断。
- **缺测试**：重复项、common AIR 合并、确定性遍历顺序。

## 6. SFF/PCX/ACT 逐函数审计

### 6.1 `SffV1Reader.ReadDirectory(string)`

- **Ikemen 语义**：通过引擎文件系统打开，可复用已加载 SFF；完整加载会建立 sprite map 和 palette list。
- **当前 C# 语义**：直接打开文件，只读目录元数据。
- **合理适配差异**：低层 reader 不做缓存是合理的；缓存应由上层资源管理器承担。
- **行为差异/冗余**：生产层没有缓存/注册表，同文件可被多次打开和解码；不能处理压缩包资源。
- **建议重构**：保留 reader，新增按规范化资源 URI 缓存的 `MSffResource`。
- **缺测试**：共享同一 SFF 的两个角色、资源释放、压缩包和只读流。

### 6.2 `SffV1Reader.ReadDirectory(Stream)`

- **Ikemen 语义**：校验头和版本，按链读取；linked sprite 复制已加载源 sprite；重复 group/number 首项生效并警告；palette 由每个 PCX 与 `same palette` 语义建立。
- **当前 C# 语义**：读取头和节点，按 `NumImages` 与文件范围封顶；只记录 linked index 和 `SamePaletteAsPrevious`。
- **合理适配差异**：目录与像素延迟解码可降低内存。
- **行为差异/冗余**：不拒绝未知版本；链提前结束时不报目录不完整；不验证 data payload；duplicate key、linked chain 和 palette 语义留给 View，且 View 未完整实现。
- **建议重构**：输出结构化 header、节点索引、诊断；建立可递归解析且有环检测的 sprite data reference。
- **缺测试**：错误版本、链环、提前结束、越界 data、重复 key、linked-to-linked。

### 6.3 `SffV1Reader.ReadNodeData`

- **Ikemen 语义**：读取失败返回错误，不把短读伪装为有效资源。
- **当前 C# 语义**：短读后返回包含尾部零值的完整长度数组。
- **行为差异**：损坏 SFF 可能被当作损坏 PCX 继续处理，错误位置被掩盖。
- **建议重构**：短读必须抛带 group/image/offset 的 `SffException`；提供 Stream 重载以复用打开句柄。
- **缺测试**：截断 payload、负/超大长度、并发读取。

### 6.4 `PcxDecoder.Decode`

- **Ikemen 语义**：SFFv1 sprite reader 处理 PCX RLE、每行跨度、palette 和上一 sprite palette 关系。
- **当前 C# 语义**：支持常见 8 位 PCX RLE，读取 256 色尾部 palette；缺失 palette 时返回全零 palette。
- **合理适配差异**：仅支持 MUGEN 常见 8 位索引 PCX 是可接受的兼容基线。
- **行为差异/冗余**：没有严格验证 manufacturer、bits-per-pixel、planes；RLE 截断时零填；把“缺 palette”和“全黑 palette”混为一谈，促使 View 用 `HasColor` 猜测。
- **建议重构**：`Palette` 缺失应为 `null` 或显式状态；严格校验格式并返回消耗字节/诊断。
- **缺测试**：无 palette、全黑但合法 palette、RLE 截断、bytesPerLine padding、多 plane 拒绝路径。

### 6.5 `SffV2Reader.ReadDirectory(string/Stream)`

- **Ikemen 语义**：读取完整 SFFv2 header、palette key 表、sprite map；支持 loaded data 与 translated data；重复 key 首项生效并警告。
- **当前 C# 语义**：读取 palette 数组和 sprite header 列表。
- **合理适配差异**：目录和像素延迟解码是合理的。
- **行为差异/冗余**：丢弃 palette group/number/numcols；没有 filename、duplicate 诊断、边界验证和统一 sprite map。
- **建议重构**：`SffV2Palette` 必须保存 key、真实色数、link、RGBA；`SffV2File` 应提供 first-wins 索引和诊断。
- **缺测试**：异常 header offset/count、重复 sprite/palette key、translated data、零 palette。

### 6.6 `ReadPalettes` / `ReadPaletteData`

- **Ikemen 语义**：palette 以 `(group, number)` 映射到物理索引，保留真实色深的 2 次幂范围（16..256）、`numcols` 与 alpha；linked palette 指向源索引；重复 key 记录警告。
- **当前 C# 语义**：每项固定扩展为 768 RGB；忽略 key、numcols 和 alpha；linked palette 共享同一 byte 数组。
- **合理适配差异**：不可变 linked palette 共享数组本身合理。
- **行为差异/冗余**：`RemapPal`、低色深 palette、SFFv2 alpha 和 `(1,n)` 选择映射无法正确实现；所有 palette 看起来都是 256 色。
- **建议重构**：使用不可变 `MPaletteData { Key, ColorsRGBA, NumColors, SourceIndex }`，palette map 与物理颜色分离。
- **缺测试**：16/32/64/128 色、alpha、linked palette、重复 key、ACT 覆盖内部槽。

### 6.7 `ReadSpriteHeaders`

- **Ikemen 语义**：保存 header 并在加载时按 link 共享像素，sprite 自身仍保留 offset、palette index 等属性。
- **当前 C# 语义**：正确区分 ldata/tdata 基址并保存常用字段。
- **合理适配差异**：数据结构简化且延迟解码合理。
- **行为差异/冗余**：没有 header offset/size 边界检查；没有保留 flags；linked sprite 链只由后续单跳处理。
- **建议重构**：保留原始 flags，构建经过验证的 data reference；linked sprite 解析需支持链和环诊断。
- **缺测试**：tdata、link chain、无效 link、超范围 payload、宽高溢出。

### 6.8 `SffV2Reader.Decode` / `DecodePixels`

- **Ikemen 语义**：支持 SFFv2 raw、RLE8、RLE5、LZ5 和 PNG 10/11/12；sprite link 共享像素但使用引用 sprite 自己的 palette/offset 语义。
- **当前 C# 语义**：支持 8 位 raw、RLE8、RLE5、LZ5；PNG 与非 8 位 raw 抛异常；linked sprite 仅解析一层。
- **合理适配差异**：逻辑程序集不依赖 PNG 库可以接受，但 PNG 解码必须在资源适配层完成，不能整体跳过。
- **行为差异/冗余**：PNG 是任意 MUGEN 1.1 角色兼容的 P0 缺口；linked-to-linked 失败；无解压长度前缀校验；palette 仍丢 alpha/色深。
- **建议重构**：统一返回 `MIndexedImage` 或 `MRgbaImage`；PNG 由可注入 decoder 处理；所有格式共享严格 bounds 与 expected-size 检查。
- **缺测试**：格式 0/2/3/4 的独立 golden vector、10/11/12 PNG、链式 link、非法压缩长度。

### 6.9 `ReadExact`

- **Ikemen 语义**：IO 错误向上传播。
- **当前 C# 语义**：短读后返回尾部零填充数组。
- **行为差异**：损坏资源被静默容忍，像素内容确定但错误。
- **建议重构**：短读抛异常；如果需要兼容容错，应显式由上层策略决定并记录 warning。
- **缺测试**：各种截断位置。

### 6.10 `Rle8Decode` / `Rle5Decode` / `Lz5Decode`

- **Ikemen 语义**：对应 `Sprite.Rle8Decode`、`Rle5Decode`、`Lz5Decode`。
- **当前 C# 语义**：位级算法按 Ikemen 结构移植，正常输入路径总体一致。
- **合理适配差异**：以 `pixelCount` 参数替代 Sprite 尺寸依赖，便于纯函数测试。
- **行为差异/冗余**：当前真实文件测试主要证明 KFM LZ5 路径，没有逐格式 golden 输出；坏输入通常零填而非形成明确诊断。
- **建议重构**：保留算法，增加消耗字节和完成标志，不要重写为另一套算法。
- **缺测试**：RLE8/RLE5/LZ5 官方或从 Ikemen 导出的逐字节 golden、边界 run、回引用越界。

### 6.11 `ActPaletteReader.Read`

- **Ikemen 语义**：最多读取 768 字节，按颜色顺序反转，index 0 alpha=0，其余 alpha=255，始终得到 256 色。
- **当前 C# 语义**：RGB 顺序反转，短文件零填；alpha 在 Unity 建图时按 index 0 单独处理。
- **合理适配差异**：RGB 与 alpha 分层存储在当前渲染模型下等价。
- **行为差异/冗余**：该函数没有 palette slot/key；ACT 结果没有被应用到 SFFv1/v2 角色资源；无法表达 palette remap。
- **建议重构**：返回 `MPaletteData`，由包加载器绑定到 `(1,palNo)`；保留原始 ACT 文件来源和覆盖策略。
- **缺测试**：短文件、超长文件、index 0 透明、应用到 SFFv1、覆盖 SFFv2 内部 palette、缺失 ACT 回退。

## 7. Unity 精灵资源逐函数审计

### 7.1 `MugenSpriteLoader.Open`

- **Ikemen 语义**：SFF、palette map、sprite map 属于角色 `CharGlobalInfo`，可按 filename 复用；角色选择 palette 后再绘制。
- **当前 C# 语义**：View 自己打开 SFF，构建 first-wins 的 group/image map，并保存 Unity Sprite cache。
- **合理适配差异**：Unity Sprite/Texture 只在表现层创建，且按动画懒加载，是正确设计。
- **行为差异/冗余**：Source 没有 `PackageId`、palette selection/remap、资源生命周期；每个 View 手工打开路径；SFF 与逻辑 `MCharData` 无强关联。
- **建议重构**：改为 `MugenVisualResourceRegistry.Open(package.VisualHandle)`，cache key 至少包含 package、sprite key、palette/remap 版本。
- **缺测试**：两个包使用同 group/image、不同行 palette、资源复用和释放、同 SFF 多实例。

### 7.2 `DetectIsV2`

- **Ikemen 语义**：统一 `SffHeader.Read` 校验签名和完整版本。
- **当前 C# 语义**：只读取 16 字节并检查 `header[15] == 2`，未检查实际读取长度和签名。
- **行为差异**：短文件可能被误判为 v1；坏签名直到后续才报错。
- **建议重构**：删除重复探测，使用统一 header reader 返回版本。
- **缺测试**：短文件、错误签名、未知版本。

### 7.3 `BuildForAnim` / `Load`

- **Ikemen 语义**：动画对象已经指向其 SFF/palette 上下文；预加载根据动画帧收集 sprite。
- **当前 C# 语义**：按 frame group/image 从当前 Source 构建 Sprite。
- **合理适配差异**：按需构建是合理的 Unity 优化。
- **行为差异/冗余**：调用接口不接收 sprite owner，因此跨角色动画只能默认使用调用者打开的 SFF；cache 不区分 palette。
- **建议重构**：参数改为 `AnimationRenderBinding`，显式带 `AnimOwnerId`、`SpriteOwnerId`、palette/remap；不要只传裸 `AnimData`。
- **缺测试**：ChangeAnim2、显式 anim/sprite owner、palette 切换后 cache 失效。

### 7.4 `BuildSpriteV2`

- **Ikemen 语义**：所有支持格式均可绘制；linked sprite 使用源像素和自身显示元数据。
- **当前 C# 语义**：PNG 直接返回 null；异常被吞掉；linked sprite 单跳。
- **行为差异**：缺图被静默隐藏，是 P0 兼容缺口且难以诊断。
- **建议重构**：不得吞掉 `SffException`；转为带资源键的兼容诊断；接入 PNG decoder。
- **缺测试**：PNG、坏 sprite 显式报告、link chain、linked sprite axis/palette。

### 7.5 `FindSharedPaletteV1`

- **Ikemen 语义**：SFFv1 palette 由每个 sprite 的 PCX palette 与 `samePaletteAsPrevious` 决定，外部 ACT 进入独立 palette list/remap。
- **当前 C# 语义**：扫描全文件，取第一张 `HasColor` 的 palette 作为全局 shared palette。
- **行为差异**：这是不规范简化。合法全黑 palette 被视为缺失；独立 palette 与 same-palette 链均错误。
- **建议重构**：删除该函数；在 SFFv1 资源导入时为每个 sprite 解析确定 palette source index。
- **缺测试**：首图无 palette、后图独立 palette、samePaletteAsPrevious、全黑 palette、ACT 覆盖。

### 7.6 `BuildSpriteV1`

- **Ikemen 语义**：link 可共享像素；sprite 使用准确 palette source 和自身 axis。
- **当前 C# 语义**：单跳 link，缺 palette 时套全局 shared palette，异常返回 null。
- **行为差异**：链式 link、独立 palette、坏资源诊断均不正确。
- **建议重构**：消费已经解析好的 `MSffSpriteResource`，View 不再推导 palette/link 规则。
- **缺测试**：linked-to-linked、自身 axis 与源像素 axis、每 sprite palette。

### 7.7 `BuildSpriteFromImage`

- **Ikemen 语义**：保留 palette alpha、sprite offset、scale、flip、混合、localcoord 和 palette remap 后绘制。
- **当前 C# 语义**：索引 0 透明、其他完全不透明；生成纹理和 pivot；`pixelsPerUnit` 由调用者决定。
- **合理适配差异**：Y 翻转与 pivot 转换是 Unity 坐标适配。
- **行为差异/冗余**：忽略 SFFv2 alpha；pixels-per-unit 与 localcoord 无绑定；每个 palette 变体需重建整张 RGBA Texture，后续 RemapPal 成本高。
- **建议重构**：短期 cache key 包含 palette；长期使用索引纹理 + palette texture/shader，与 Ikemen palette 模型更接近。
- **缺测试**：pivot、非 320 localcoord、SFFv2 alpha、palette 切换、flip/offset 合成。

### 7.8 `HasColor`

- **Ikemen 语义**：palette 是否存在由文件结构决定，不由颜色值猜测。
- **当前 C# 语义**：全部 RGB 为零即判定 palette 不存在。
- **行为差异**：合法全黑 palette 被错误替换。
- **建议重构**：删除颜色启发式，使用显式 `HasEmbeddedPalette`。
- **缺测试**：全黑 palette。

### 7.9 `MugenSpriteAnimator.Key` / `Play` / `Update` / `Apply`

- **Ikemen 语义**：动画由逻辑帧推进，动画对象携带 SFF/palette/owner/local scale；渲染读取当前 frame。
- **当前 C# 语义**：MonoBehaviour 使用 `Time.deltaTime` 自行推进另一套 `AnimAdvance`，按裸 group/image 查 Sprite。
- **合理适配差异**：独立动画展示组件可以自行计时。
- **行为差异/冗余**：用于实际战斗时会和 `MAnimSystem` 形成双时钟；没有 offset、vertical flip、palette、scale、owner 和 localcoord；缺 sprite 时保留上一张图而非明确空帧。
- **建议重构**：战斗 View 只能消费逻辑快照的 `AnimNo/Elem/ElemTime`；展示组件与战斗组件分离命名；`Apply` 消费完整 render binding。
- **缺测试**：逻辑帧同步、空帧清图、offset/flip、ChangeAnim2 sprite owner、暂停/回滚重播。

## 8. SND 与声音所有权逐函数审计

### 8.1 `SndFile.TryGet` / `Key`

- **Ikemen 语义**：SND 以 `(group, number)` map 查询。
- **当前 C# 语义**：用 64 位组合键查目录。
- **合理适配差异**：等价且确定性良好。
- **行为差异/冗余**：`SndFile` 不保存 filename/package owner，不能独立用于资源复用和诊断。
- **建议重构**：组合键可保留；SndFile 由上层 `MSoundBankResource` 包装来源与 package ID。
- **缺测试**：负数、极值、键碰撞证明。

### 8.2 `SndReader.Read`

- **Ikemen 语义**：`LoadSnd` 可复用同 filename；`LoadSndFiltered` 默认只保留 group/number 均非负项；重复 key 首项生效并警告；读取 WAV 并验证，坏声音禁用但可继续加载。
- **当前 C# 语义**：读取全部目录项和 payload 范围；重复 key 首项生效；额外检测 subheader 环。
- **合理适配差异**：只建立目录、延迟复制 WAV，适合 Unity 侧懒解码；环检测比参考实现更严格，是合理增强。
- **行为差异/冗余**：负 group/number 也进入表；不验证 WAV；`next=0` 但 count 未结束时会在 offset 0 继续并最终报错；没有 filter/max 和资源复用。
- **建议重构**：reader 接收过滤策略；默认角色 bank 对齐 Ikemen 的非负过滤；每条 entry 保存 valid/diagnostic；缓存由资源注册表承担。
- **缺测试**：负 key、duplicate warning、坏 WAV、提前终链、环、filter/max、超大 count。

### 8.3 `SndReader.CopyWave`

- **Ikemen 语义**：`readSound` 在加载时解析 WAV，播放端取得可用 streamer。
- **当前 C# 语义**：按 entry 偏移复制原始 WAV 字节。
- **合理适配差异**：Unity `AudioClip` 解码推迟到表现层合理。
- **行为差异/冗余**：没有再次验证 entry/source 对应关系；每次复制分配；生产端没有消费者。
- **建议重构**：提供只读切片/stream；Unity adapter 按 bank+key 缓存 AudioClip；解析失败进入兼容报告。
- **缺测试**：错配 source、截断、缓存复用、实际 WAV 解码。

### 8.4 `PlaySndController.Run`

- **Ikemen 语义**：声音 bank 可由 `S/F/其他 FX 前缀` 选择；相对 pan 使用角色位置、朝向和 `localscl`；freqmul 限制 0.01..5；WinMUGEN loop/volume 有版本兼容；channel、priority、stop flags 由声音通道系统执行。
- **当前 C# 语义**：发出确定性 `MSoundEvent`，记录大多数参数；`CommonBank` 仅区分 F 与非 F；不执行声音设备。
- **合理适配差异**：模拟只发事件是正确架构，音频不能进入锁步核心。
- **行为差异/冗余**：事件没有 `SoundOwnerResourceId`，只有实体 `OwnerId`；没有状态 owner/FX prefix；相对 pan 没有位置与 localcoord 上下文；freqmul 不 clamp；WinMUGEN 兼容规则缺失；无人消费事件。
- **建议重构**：事件在产生时解析成稳定 `SoundBankKey`，携带 owner package ID、位置或已标准化 pan、规则版本；增加 Unity audio adapter。
- **缺测试**：helper、custom state、F/S bank、位置 pan、facing、freq clamp、loop volume、channel priority、真实播放 adapter。

### 8.5 `StopSndController.Run` / `SndPanController.Run`

- **Ikemen 语义**：操作角色拥有的声音通道；StopSnd `-1` 有全角色声音兼容行为；SndPan 相对 pan 乘 facing 并使用位置/local scale。
- **当前 C# 语义**：仅发 `OwnerId + Channel` 事件，pan 保留原始参数。
- **合理适配差异**：事件化正确。
- **行为差异/冗余**：通道所有权、`-1` 全局语义、相对 pan 坐标、redirect owner 均未闭环。
- **建议重构**：通道 key 使用 `AudioOwnerId + Channel`，明确 global stop 事件；在逻辑事件或表现 adapter 中统一计算相对 pan，不能两边各猜。
- **缺测试**：`channel=-1`、redirect/helper、多个角色同 channel、相对/绝对 pan。

## 9. localcoord 逐函数与运行链审计

### 9.1 `MCharacterDefinition.LocalCoordWidth/Height`

- **Ikemen 语义**：角色 global info 默认 320x240；加载 DEF 后初始化角色 `localcoord`、`localscl` 和 portrait scale。
- **当前 C# 语义**：仅保存两个整数。
- **结论**：数据模型存在，但运行语义未实现。

### 9.2 `MConstants.Read`

- **Ikemen 语义**：大量 size/velocity/movement 常量在读取 trigger 或应用时乘 `((320 / c.localcoord) / stateOwner.localscl)` 一类比例；自定义状态会切换状态坐标解释。
- **当前 C# 语义**：直接返回原始固定点常量；文件注释明确假设“v1 单坐标 320，故略”。
- **行为差异**：这不是任意角色兼容下可接受的简化。640 localcoord 角色会以约两倍尺度运行；不同 localcoord 的投技状态更严重。
- **建议重构**：常量保持原始值；`Read` 或表达式上下文接收 `MCoordinateContext` 后按字段类别缩放。禁止导入时统一除二，因为状态 owner 可在运行时变化。
- **缺测试**：640 与 320 等价角色的 const、速度、重力、尺寸对照；跨 localcoord 自定义状态。

### 9.3 `MPhysics.Step`

- **Ikemen 语义**：速度、重力和位置在当前角色/状态比例下解释；角色位置与舞台比例互相转换。
- **当前 C# 语义**：直接积分固定点速度并直接加原始 `Yaccel`；注释同样声明忽略 localcoord。
- **行为差异**：非 320 角色的运动轨迹不兼容。
- **建议重构**：明确内部世界坐标单位，例如统一 320 基准；所有 controller 参数和常量通过当前状态坐标上下文转换后再写入内部值。
- **缺测试**：320/640 相同动作轨迹、跳跃重力、摩擦阈值、舞台起点换算。

### 9.4 `AirParser.ParseBox/ParseFrame` 与 `MAnimImport`

- **Ikemen 语义**：AIR 数值属于动画定义所有者坐标；切换动画 owner 后更新 `animlocalscl` 和碰撞缩放；sprite owner 不同时另有 sprite scale 比例。
- **当前 C# 语义**：Clsn、offset 原样固定点化；运行时无动画 owner 坐标信息。
- **行为差异**：跨 localcoord 的 ChangeAnim2，画面、Clsn 和命中位置会分离。
- **建议重构**：静态 AIR 保留原始坐标；运行时生成 `AnimRenderContext` 与 `CollisionTransform`，由 anim owner/localcoord 计算。
- **缺测试**：640 owner 的动画给 320 被投角色、sprite owner 保持自身时的碰撞与画面比例。

### 9.5 `MugenSpriteLoader.BuildSpriteFromImage`

- **Ikemen 语义**：动画 local scale、sprite owner localcoord、角色自身 localcoord 共同决定绘制比例。
- **当前 C# 语义**：只使用调用者给出的固定 `pixelsPerUnit`。
- **行为差异**：同一 SFF 在不同角色坐标和跨所有权动画下比例错误。
- **建议重构**：纹理资源本身不烘焙 localcoord；Renderer transform 根据 `RenderBinding` 计算 scale。
- **缺测试**：非 320、跨 owner、不同舞台分辨率。

## 10. 动画/精灵所有权逐函数审计

### 10.1 `MChar.AnimTable`

- **Ikemen 语义**：静态表在 `cgi[playerNo].animTable`；运行角色仅用 `animPN` 指向当前表所有者。
- **当前 C# 语义**：角色直接保存字典引用，既代表自身表，又代表当前活动表。
- **行为差异**：静态身份与运行选择混为一体。
- **建议重构**：移除可写 `AnimTable`；改为只读 `SelfResourceOwnerId` 和可回滚的 `AnimOwnerId/SpriteOwnerId`。
- **缺测试**：所有权切换与恢复、clone/hash、多个同包实例。

### 10.2 `MChar.AnimExists`

- **Ikemen 语义**：`animexist` 查 `animPN` 当前动画表；`selfanimexist` 永远查自身 `gi().animTable`，两者可在同一状态中返回不同结果。
- **当前 C# 语义**：二者统一调用当前 `AnimTable`。
- **行为差异**：明确不兼容，且当前注释承认该简化。
- **建议重构**：实现两个查询：`AnimExists` 经 `AnimOwnerId`，`SelfAnimExists` 经 `SelfResourceOwnerId`。
- **缺测试**：ChangeAnim2 后两个 trigger 返回不同值；无效 owner 回退。

### 10.3 `MChar.CanChangeAnimTo`

- **Ikemen 语义**：`ChangeAnim` 默认查自身 owner；`ChangeAnim2` 查状态 owner；显式 anim player 参数可指定其他 owner；无效动画不切换。
- **当前 C# 语义**：只检查当前可变 `AnimTable`，无表时甚至放行。
- **行为差异**：ChangeAnim2 污染后，普通 ChangeAnim 查错表；测试构造的 null 表行为与真实引擎不同。
- **建议重构**：该函数必须接收 `ownerId` 和 change kind；生产角色不允许 null resource registry，测试用显式 empty package。
- **缺测试**：ChangeAnim2 后普通 ChangeAnim、无效动画、`-1/-2/负数`。

### 10.4 `ChangeAnim2Controller.Run`

- **Ikemen 语义**：默认动画 owner 为状态 bytecode 的 `playerNo`，sprite owner 固定为角色自身；成功后更新 owner 与 localcoord/clsn；支持 readplayerid；不修改静态表。
- **当前 C# 语义**：从 `StateOwner.AnimTable` 取表，并把 `character.AnimTable` 永久改成该表；sprite owner 隐式依赖 View 当前打开的 SFF。
- **行为差异**：这是当前模块最严重的实现问题。`SelfState` 只清 `StateOwner`，不会恢复 `AnimTable`；之后普通动画、trigger、clone 都可能继续使用对方表。也无法表达 anim owner 与 sprite owner 分离、readplayerid、FFX、无效 owner 回退和动画 owner localcoord。
- **建议重构**：控制器只调用 `ChangeAnimation(animNo, animOwnerId, selfSpriteOwnerId, elem, elemTime)`；owner 由状态执行上下文提供，不从可变对象引用推断。
- **缺测试**：投技 ChangeAnim2 -> SelfState -> ChangeAnim；双方同 action number 不同内容；不同 localcoord；readplayerid；无效 action。

### 10.5 `MAnimSystem.Action` / `Play` / `PlayAt`

- **Ikemen 语义**：角色保存当前 `Animation` 实例，实例已绑定动画定义、SFF/palette context；owner 变化同步更新比例和当前 frame。
- **当前 C# 语义**：每次由外部 table+AnimNo 查静态数据，运行计数在 `MChar`；`PlayAt` 不记录 table owner。
- **合理适配差异**：运行计数拆入 `MChar` 有利于回滚，是合理的 C# 适配。
- **行为差异/冗余**：系统无法验证传入 table 与 `MChar.AnimTable`/View SFF 是否一致；`AnimRunningNo` 只用 action number，两个 owner 都有同号动画时 owner 切换不会自然成为独立身份。
- **建议重构**：运行标识改为 `AnimationKey { OwnerId, ActionNo }`；`Action` 自行通过 registry 解析，调用者不传裸 dictionary。
- **缺测试**：同 action number 跨 owner 切换、owner 切换重置、rollback、无效 action 保持旧动画。

### 10.6 `MChar.Clone` / `WriteHash`

- **Ikemen 语义**：owner 字段属于运行状态，存档/回滚必须恢复；静态资源本身共享。
- **当前 C# 语义**：浅拷 `AnimTable`；哈希包含 `StateOwner.Id`，不包含动画/精灵 owner，因为不存在独立字段。
- **合理适配差异**：静态资源不进入 hash 正确。
- **行为差异/冗余**：字典引用成为隐含运行状态，hash 无法识别两个不同 owner 的同内容/同 action；重链只处理 `StateOwner`，无法恢复资源身份。
- **建议重构**：hash/clone 只记录 owner ID 和 action key，资源表仍由 registry 共享。
- **缺测试**：owner 差异导致不同 hash、快照后资源重链、同包多实例。

### 10.7 `MugenSpriteLoader` 与 `MugenSpriteAnimator` 的隐式 sprite owner

- **Ikemen 语义**：`animSpriteSetup` 显式把动画绑定到 `spritePN` 的 SFF/palette，并按 owner localcoord 调整 scale。
- **当前 C# 语义**：哪个 View 打开了哪个 SFF，就隐式决定 sprite owner。
- **行为差异**：简单 ChangeAnim2 恰好可能显示“自己的精灵”，但这只是调用路径巧合；显式 `spriteplayerno`、FX、helper/custom state 和多个包无法可靠表达。
- **建议重构**：所有渲染请求必须带 `SpriteOwnerId`，不得从 GameObject 或当前 Source 推断。
- **缺测试**：动画 owner A + sprite owner B、palette owner B、owner localcoord 比例。

## 11. 推荐目标架构

建议一次建立以下边界，不再增加双表补丁：

```text
MCharacterPackage
  PackageId
  Manifest                 // DEF、来源、编码、已解析资源 URI、诊断
  LogicData                // states、commands、constants
  AnimationLibrary         // 原始 AIR + 逻辑动画 + 表现动画
  VisualResources          // SFF、sprite map、palette bank
  SoundResources           // SND bank
  CoordinateSpace          // localcoord、portrait scale

MResourceRegistry
  ResolvePackage(packageId)
  ResolveAnimation(ownerId, actionNo)
  ResolveSprite(ownerId, group, image, paletteContext)
  ResolveSound(ownerId, commonBank, group, number)

MChar runtime
  SelfResourceOwnerId
  StateOwnerId
  AnimOwnerId
  SpriteOwnerId
  CurrentAnimationKey(ownerId, actionNo)
  CoordinateContext(self/state/anim/sprite)
```

设计约束：

1. 静态资源不可变，不能通过赋值改变角色的“自己的资源”。
2. owner 用稳定 ID，不用对象引用或字典引用表达。
3. 状态 owner、动画 owner、精灵 owner、声音 owner 分开。
4. `localcoord` 在运行时控制参数解释；不能只在导入时整体缩放。
5. Logic 不创建 Unity 对象和 AudioClip，但事件必须携带可唯一解析资源的 key。
6. 所有兼容容错必须形成结构化 diagnostic，不能吞异常或静默返回 null。
7. View 只消费逻辑帧状态和资源 key，不运行第二套战斗动画时钟。

建议迁移顺序：

1. 建立 `PackageId + MResourceRegistry`，把现有 `MCharData` 包装为完整 package。
2. 增加四类 owner ID 和 `AnimationKey`，替换 `ChangeAnim2` 的字典赋值。
3. 接入 `MCoordinateContext`，先覆盖常量、controller 参数、AIR Clsn/offset、物理积分。
4. 完整重做 SFF palette model，并支持 PNG 10/11/12。
5. 将 ACT/SND 纳入包加载，接 Unity palette/audio adapter。
6. 最后补 AIR Copy Action、插值和完整表现属性。
