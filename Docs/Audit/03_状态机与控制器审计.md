# 03 状态机与控制器审计

## 1. 审计范围与基线

- 本文只审计 `MugenCnsParser`、`MStateDef`、`MStateMachine`、`Assets/Logic/Mugen/StateCtrl/*.cs` 以及它们直接依赖的状态所有权执行链。
- 对照版本：Ikemen GO commit `843f732`。
- Ikemen 主要基线：`src/compiler.go`、`src/compiler_functions.go`、`src/bytecode.go`、`src/char.go`。
- 本文基于 2026-06-06 工作树现状。工作树已有其他未提交修改；本审计未改运行代码。
- 评级：`一致`、`合理适配`、`部分实现`、`危险简化`、`未实现`。

结论不是“目前能跑 531 个测试，所以可以导入任意角色”。当前测试证明的是已有样例路径可运行；状态编译、所有权、运行期参数和静默降级仍存在会直接改变状态机结果的差异。

## 2. 执行摘要

### P0：会改变状态机与招式结果

1. **HitDef/Projectile/ReversalDef 参数被解析期固化。** `BuildHitDef` 用 `Compile(...).Run(null)` 立即求值。Ikemen 把参数编译为表达式，在控制器每次执行时以当前角色 `c` 求值。任何使用 `var`、`fvar`、`random`、`time`、重定向或其他运行态 trigger 的 HitDef 都会错误。
2. **自定义状态的负状态所有权错误。** Ikemen 的 `-3/-2/-1` 始终取被控制角色自己的 `gi().states`，并按 `state owner == self`、`keyctrl`、player/helper 类型门控。当前 `MBattleEngine.StateDataFor` 把整张状态表切到 `StateOwner`，随后 `MStateMachine` 直接从这张外国表运行 `-3/-2/-1`。
3. **自定义状态中的 `SelfState` 同帧查错状态表。** `ApplyTransition` 清掉 `StateOwner`，但仍使用本次 `RunFrame` 传入的外国 `states/commonStates`；状态头初始化和同帧重入仍可能执行外国同号状态。Ikemen `selfState` 明确以自身 `playerNo` 调 `changeStateEx`。
4. **`TargetState` 没有传递状态所有者。** Ikemen `targetState` 使用当前工作状态的 `sb.playerNo` 调目标的 `stateChange1`。当前只写目标的 `PendingStateNo`，不把当前 state owner 赋给目标。
5. **未知/无效控制器静默变成合法 `NullController`。** Ikemen 对 unknown controller、缺失 `type`、缺失 `trigger1` 返回编译错误；当前默认 `Parse(text)` 没有报告对象，unknown、缺失 type、部分无效 Var 控制器均可被静默丢弃或 no-op。
6. **`ParameterOnlyController` 把未实现功能伪装成成功执行。** `ReversalDef`、`AttackDist`、Tag、PalFX 等都能通过解析并参与 trigger/persistent，但 `Run` 无效果。对导入器而言这是错误的成功状态。
7. **`DestroySelf` 不终止当前控制器链。** Ikemen `destroySelf.Run` 在销毁自身时像 ChangeState 一样返回 true；当前仅设置 `Destroyed=true` 并返回 false，后续控制器仍执行。
8. **HitDef 每次执行清空 `Targets`。** Ikemen 重置并 finalize HitDef，不等价于清空 target ownership。当前 `HitDefController.Run` 的 `c.Targets.Clear()` 会破坏 HitOnce、Target 系列和投技链。
9. **persistent 的存储模型会碰撞。** `(stateNo * 256 + ctrlIndex)` 在单状态超过 256 个控制器时碰撞；负状态之间也可能因大索引互相覆盖。Ikemen 为每个 `StateBytecode` 保存独立 `ctrlsps[]`。
10. **命中暂停中的状态切换不兼容。** Ikemen 在 hitpause 中延迟 `stateChange2/init`，并保留/修正 persistent 计数与 `hitStateChangeIdx`；当前立即 `ApplyTransition + RunInit + 同帧重入`。

### P1：当前样例已实际使用，但结果不完整

对 `D:/Desktop/demo/MugenSource` 的静态统计显示：`PlaySnd 1817`、`Explod 1256`、`PalFX 311`、`EnvShake 199`、`AfterImage 189`、`ScreenBound 115`、`Width 111`、`HitOverride 48`、`ReversalDef 11`、`AttackDist 6`。其中 PalFX/AfterImage/EnvShake 是表现缺口；ScreenBound/Width/PlayerPush/HitOverride 当前基本是 write-only；ReversalDef/AttackDist 会改变战斗逻辑。

## 3. CNS Parser 逐函数审计

| 函数 | 评级 | 与 Ikemen `843f732` 的差异及影响 |
|---|---|---|
| `Parse(string)` | 危险简化 | 默认不传 `MCompatibilityReport`，unknown 与 parsed-only 信息完全丢失。导入任意角色时无法判断“成功”是否真实。 |
| `Parse(string, report)` | 危险简化 | 单遍文本扫描可用于受控 MUGEN 子集，但缺少源位置、文件名、错误级别和严格模式；重复 Statedef 直接最后覆盖。Ikemen 对普通重复状态采用先声明者，并对负状态/common 有专门覆盖规则。 |
| section/header 处理 | 部分实现 | 只接受 `statedef` 和 `state`；未知段直接清空 current。畸形 header、无等号行均静默忽略。Ikemen 会形成带文件/行号的编译错误。 |
| `ApplyStatedefField` | 危险简化 | 仅支持 type/movetype/physics/ctrl/anim/facep2/juggle/poweradd/velset；缺 `hitcountpersist`、`movehitpersist`、`hitdefpersist`、`sprpriority`。未知字段静默忽略。 |
| `FlushController` | 危险简化 | 没有 type 时整个块被静默丢弃；Ikemen 报 `State controller type not specified`。`persistent/ignorehitpause` 只保存 int/bool，丢失 Ikemen block 的 tri-state 和 hitpause ChangeState 特例。 |
| `FlushController` parsed-only 检测 | 部分实现 | 通过反射判断是否继承基础 no-op，能统计一部分未实现项；但报告是可选的，而且 write-only 控制器不会被标为 parsed-only。 |
| `BuildTriggers` | 危险简化 | triggerall AND、同号 trigger AND、组间 OR 的核心正确；但允许无 `trigger1` 时恒真，也允许 `trigger3` 跨过缺失的 trigger1/2。Ikemen 默认严格报错。 |
| `BuildController` | 危险简化 | 工厂集中可读，但 unknown 返回 `NullController`；没有统一 required-param 验证、redirectid 编译和能力标记。标准 MUGEN 控制器还缺 `ExplodBindTime`、`Zoom`。 |
| `BuildRelayVar` | 危险简化 | 无效 Var 形态再次退化为 `NullController`，与合法 `type = Null` 无法区分。 |
| `BuildVarSet` | 部分实现 | 支持常见 `v/fv/var()/fvar()`；index 被解析成字面量，缺 Ikemen 的运行期 index、重定向和变量范围语义。字典允许任意 index，掩盖越界角色错误。 |
| `BuildHitBy` | 危险简化 | attr/time 被固化，只有单一全局槽；缺 slot、stack、playerno、playerid、redirectid。Ikemen 有多槽 HitBy 记录。 |
| `BuildHitDef` | 危险简化 | 最大缺口。绝大多数参数通过 `EvalI/EvalF(...Run(null))` 在导入时求值；Ikemen 在每次 `HitDef.Run` 时求值并 reset/finalize。字段覆盖也只是 HitDef 全集的一小部分。 |
| `FillPowerDefaults` | 部分实现 | 用固定 0.7/0.6/0.5 推导，未读取角色/引擎 constants 与版本行为；动态 damage/getpower/givepower 无法成立。 |
| `TruncMul` | 合理适配 | 用整数除法模拟向零截断，适合锁步；前提是输入默认值本身正确。 |
| `ParseVel` | 危险简化 | 立即求值且只取 XY，丢运行期表达式、Z、localcoord/redirect 缩放。 |
| `ParseReaction` / `ParseHitType` | 部分实现 | 常见枚举可用；未知值默认 Light/High，未产生诊断。 |
| `EvalI` / `EvalF` | 危险简化 | `Run(null)` 是当前 parser 最核心的架构错误。编译结果应保留到控制器执行期。 |
| `ParseFlags` / `FlagCode` | 部分实现 | 支持当前枚举子集；unknown flag 静默为 0，Ikemen 的 AssertSpecial 标志集更大。 |
| `Expr` / `ExprList` | 部分实现 | 保留运行期表达式的方向正确；用裸 `Split(',')` 不能可靠处理字符串、嵌套参数和复杂语法中的逗号。 |
| `WithParams` | 不规范冗余 | 同时保存 raw `Dictionary<string,string>` 和 typed expression 字段，产生双重事实源。运行代码只用 typed 字段，raw 参数主要用于“看似已解析”。 |
| `ParseHitFlagMask` | 部分实现 | 只保留 H/L/A/D 四位，无法表达 Ikemen 完整 guard/reversal flag 语义。 |
| `FillPalFX*` | 部分实现 | 参数采集较全，但对应控制器全是 no-op；属于“解析成功、执行失败”。 |
| `FillAfterImage*` | 部分实现 | 同上；`ignorehitpause` 还与控制器通用 ignorehitpause 混在同一参数表，运行期没有独立效果。 |
| `FillPlaySnd` | 部分实现 | S/F bank 前缀和常用字段已编译；字符串切分仍脆弱，声音 owner/channel 生命周期、stop-on-state-change 依赖后续系统。 |
| `FillExplod` / `FillExplodInterpolation` | 部分实现 | 采集字段远多于 `ExplodController.ApplyToExplod` 实际消费字段，形成大量无效参数。 |
| `FillText` | 未实现 | 只采集参数，Text/ModifyText/RemoveText 均无执行实体。 |
| `Get` / `IntP` / `ParseFirstInt` | 危险简化 | `IntP` 被 ChangeState/SelfState 的 ctrl/anim 使用，导致本可为表达式的参数退化为字面量，解析失败时静默回默认值。 |
| `ExtractParenInt` | 部分实现 | 只接受字面量 index，失败回 0，可能写错 `var(0)`。 |
| `ParseTrailingInt` | 危险简化 | 畸形 Statedef 回 0，可能覆盖合法 state 0；应返回诊断而非默认。 |
| `StripComment` | 部分实现 | 第一个 `;` 后全部删除，不识别字符串上下文。Clipboard/Text 字符串内分号会被截断。 |

## 4. MStateDef 与 MStateMachine 逐函数审计

### `MTriggerSet` / `MStateDef`

| 函数 | 评级 | 差异及影响 |
|---|---|---|
| `MTriggerSet.Passes` | 合理适配 | 对普通 CNS 的 triggerall AND、组内 AND、组间 OR 正确。问题来自 parser 接受缺失 trigger1 和表达式编译器的 unknown 降零。 |

| `SelfStateController.Run` | 危险简化 | 参数缺口同上；更严重的是实际 transition 使用旧的外国 states 参数，不能保证同帧回自身状态表。 |

### `MStateMachine`

| 函数 | 评级 | 差异及影响 |
|---|---|---|
| `RunFrame` | 危险简化 | 顺序接近 actionRun 主干，但缺 state `-4`、`-10`，缺 helper/keyctrl/owner 门控。外部 pending transition 在负状态之前立即应用，Ikemen 在 actionRun 中先运行负状态再 `stateChange2`。 |

| `RunCurrentState` | 部分实现 | 平坦 controller 链和同帧重入方向正确；`MaxReentry=16` 显著低于 Ikemen `MaxLoop=2500`，合法长链会提前截断且无诊断。 |

| `PersistGate` | 部分实现 | 基本倒计时顺序一致；缺 hitpause ChangeState correction 与 per-StateBytecode 独立数组。 |
| `ResetPersist` | 部分实现 | 普通未切状态路径一致；负状态 ChangeState 路径调用时机错误。 |
| `ClearStatePersist` | 危险简化 | 以状态号范围删除，既受 stride 碰撞影响，也不能区分同号的 own/common/foreign StateBytecode。 |
| `ApplyTransition` | 危险简化 | pending state 不携带 owner id、anim owner、source state context。SelfState 清 owner 后仍使用旧 states；无 invalid-state 诊断、localcoord 切换、hitpause 延迟和 explod/sound state-change side effects。 |
| `Lookup` | 部分实现 | 简单 own-first/common fallback 可用；Ikemen 在编译/装载阶段完成更细的 common/duplicate/negative state 归并，运行期不是两个裸字典回退。 |

## 5. StateController 工厂与执行逐项审计

## 6. 设计判断：合理适配与不规范简化

### 可以保留的合理设计

1. `MTriggerSet` 用显式列表表示 triggerall/trigger groups，比照搬 Go bytecode block 更符合 C# 可测试性。
2. 固定点 `FFloat`、共享确定性 RNG、事件化 PlaySnd、逻辑 Explod record 都符合锁步/rollback 目标。
3. `MCharData` 保存静态定义、`MChar` 保存运行态的总体方向正确。
4. StateController 使用小类而非一个巨型 switch，本身不是过度设计，便于逐控制器测试。

### 应重构的不规范设计

1. **删除 `ParameterOnlyController` 的“可执行 no-op”语义。** Unsupported node 必须带 controller name/source span/capability，并在 strict import 下失败；合法 Null 应保持独立。
2. **引入不可变编译 IR。** parser 只负责语法和表达式编译；HitDef/Projectile 等保存 `BytecodeExp` 参数，不保存导入期求值后的模板。
3. **建立 `StateExecutionContext`。** 至少包含 executing char、state owner player id、resource owner ids、minus state、hitpause mode、source StateBytecode identity。不要再用 `MChar StateOwner + 可变 AnimTable` 推断。
4. **Transition 必须是原子值对象。** `PendingTransition { stateNo, ownerId, selfState, animExp/result, ctrl, continue }`，由 transition executor 统一校验、切 owner、init、重入。
5. **persistent 改为每个编译 StateDef 的数组。** 不使用 `stateNo*256` 魔数，也不能跨 own/common/foreign 同号状态共享。
6. **负状态调度从普通 Lookup 分离。** 按 Ikemen actionRun 明确实现 -4/-3/-2/-1/-10、player/helper、keyctrl、state owner 和 pause 门控。
7. **统一 controller 公共参数。** `redirectid`、trigger、persistent、ignorehitpause、source span 应由 controller block 基类/执行器处理，不在 90 多个类中遗漏。
8. **把 write-only 状态视为未实现。** Width/PlayerPush/ScreenBound/HitOverride 只有字段没有消费者，不能标“完成”。
9. **解析必须有 strict 与 compatibility 两种模式。** strict 对 unknown、缺 type、缺 trigger1、必填参数、表达式错误直接失败；compatibility 可降级但必须输出结构化报告并阻止“完整适配”认证。

## 7. 建议修复顺序

1. 先重构 parser IR，移除 `Run(null)`，补 source diagnostics 和 strict mode。
2. 建立 state owner/transition context，修复 custom state、SelfState、TargetState、ChangeAnim2。
3. 重写 negative-state scheduler 与 per-StateBytecode persistent，补 hitpause transition correction。
4. 修复 HitDef/HitBy/HitOverride/ReversalDef/AttackDist/Target bookkeeping；这是招式能否正确切状态的核心。
5. 补 Helper/Projectile 完整运行期参数和 spawn 时序。
6. 把 Width/PlayerPush/ScreenBound 接到实际 solver；否则保持 unsupported，不要 write-only。
7. 最后接表现层 PalFX/AfterImage/EnvShake/Trans/Angle/Text，并通过 trace 分离 gameplay exact 与 presentation event exact。

## 8. 审计判定

当前模块适合作为**受控角色子集的原型**，不满足“导入任意 MUGEN 角色即可完整适配”的工程定义。主要问题不是代码量少，而是解析成功、执行成功和兼容成功三个概念没有在类型与导入流程中分开。继续逐个补字段会扩大补丁式架构；应先收口编译 IR、状态所有权、transition、persistent 和 unsupported capability，再扩展控制器覆盖。
