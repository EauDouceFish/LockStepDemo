## 2. 总体结论

当前代码不是“任意 MUGEN 角色完整适配”的命中与实体实现。它能够支撑受控样例中的单次普通攻击、基础受击状态、最小 helper/projectile/explod 记录和简化回合流程，但核心语义被压缩得过多。

合理的工程适配有三项：

1. 用 `FFloat` 替换浮点运行态，适合确定性锁步。
2. 用 `MEntityWorld` 代替 Ikemen 全局实体数组，方向正确。
3. 把帧内伤害延迟到统一阶段提交，方向正确，但当前提交模型不等价于 Ikemen。

主要问题不是代码量少，而是领域模型缺失：运行期 HitDef 表达式、hit id、接触结果、GetHitVar 累积规则、暂停时钟、实体所有权和回合清理都没有成为一等状态。继续在 `MHitSystem.TryHit` 和 `MBattleEngine.Tick` 中追加条件会形成补丁架构。

### 判定标签

| 标签 | 含义 |
|---|---|
| A 合理适配 | 与 Ikemen 离散行为等价，差异仅来自定点化、无 UI 或依赖注入 |
| B 有条件合理 | 结构方向正确，但只覆盖明确子集；扩大兼容范围前必须补模型 |
| D 差异性简化 | 样例可运行，但会让真实角色产生可观察差异 |
| X 错误/不兼容 | 已与 Ikemen 语义冲突，现有测试可能固化了错误行为 |

## 3. P0 缺口

P0 是进入“任意角色 trace 对账”前必须修复的语义基础。

| `ModifyExplodController.Run` | B | selector + 批量修改结构合理；应区分 selector 参数与可修改参数 |

explod 虽多为表现实体，但 `numexplod`、`explodvar` 和 remove 条件参与状态机，所以其生命周期属于模拟层 P0，不可全部推给 Unity 表现层。

### 4.8 Pause、SuperPause 与 hitpause

| 当前函数 | 判定 | Ikemen 差异与结论 |
|---|---|---|
| `MPauseState.Step` | B | buffer 下一帧应用和 super 优先方向接近 Ikemen；缺 pausebg、endcmdbuftime 等全局状态 |
| `MPauseState.Clone/WriteHash` | A | 对当前字段完整 |

## 8. Trace 收口要求

本模块不能只比较每帧最终 hash。需要生成结构化 hit/entity trace，才能定位第一处分歧。

每帧至少记录：

1. `round`: roundState、intro/over timer、round timer、winner、wins、draws。
2. `pause`: pause/super time、buffer、playerNo、每实体 pauseMovetime/superMovetime/hitPause/unhittable、execution mask。
3. `hitdef-eval`: source entity/state/controller、完整运行期 HitDef 字段摘要。
4. `collision-candidate`: attacker/defender/projectile id、双方 frame/clsn、坐标空间、overlap 结果。
5. `eligibility`: affectteam、hitflag、HitBy、HitOverride、ReversalDef、juggle、chain/nochain、unhittable 各门禁结果。
6. `contact`: hit/guard/reversal/cancel/trade、priority、hit id、target buffer 变化。
7. `ghv-before/after`: damage、hitcount、guardcount、hitid、playerid/projid、fall、vel、timers、targetedBy。
8. `damage-commit`: 每个来源的 raw/scaled damage、kill、累计值、提交前后 Life。
9. `state-transition`: old/new state、state owner、HitOverride/guard/default 路由原因。
10. `entity`: helper/projectile/explod spawn、tick、status、owner、remove reason、创建顺序。

对账规则：

- entity id 可建立映射，不要求数值与 Ikemen 相同，但 owner/创建顺序/关系必须一致。
- 固定点连续量允许配置容差；状态号、接触类型、计数、目标集合、生命、时钟必须精确一致。
- 第一个离散分歧立即停止并输出该帧全部门禁，不允许只报最终 hash 不同。
- 每个 P0 测试都要能导出同一 trace schema；不能另造测试专用观测路径。

## 9. 推荐实施顺序

1. 先完成 HitDef spec/runtime、GetHitVar、hit id/targetedBy 数据模型。
2. 再拆分 contact pipeline，并接入 HitOverride/ReversalDef/guard/juggle。
3. 接入统一 execution mask，修复 hitpause/pause。
4. 将 helper/projectile/explod 改成 owner-scoped、id-based 生命周期。
5. 建立集中式 round context 和 next-round reset。
6. 最后接 Ikemen 双端 trace；每完成一个阶段即冻结对应 golden trace。

在 P0-01 至 P0-20 完成前，不应宣称“任意 MUGEN 角色命中与实体系统完整适配”。当前最危险的做法是继续补单个 controller 字段，因为根本差异集中在运行期求值、接触时序和所有权模型，而不是缺少几个 if。
