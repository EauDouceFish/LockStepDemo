# 表达式 VM、Trigger、Redirect 与命令系统审计

## 1. 审计边界与结论

- 审计对象：`Assets/Logic/Mugen/Expr`、`MChar.ReadTrigger/Redirect`、`MTriggerSet`、`Assets/Logic/Mugen/Command`、`MugenCmdParser`，以及仍留在工程中的旧 `Game/Expr`、`Game/Command` 管线。
- 对照基准：Ikemen GO commit `843f7327efda98f6d47edbf38f45420b88cee3c8`，主要对应 `src/bytecode.go`、`src/compiler.go`、`src/input.go`。
- 本文只审计，不修改运行代码。

总体判断：当前代码已具备可运行的“核心子集”，但还不是 Ikemen/MUGEN 语义兼容层。主要问题不是定点数，而是编译器静默降级、trigger 覆盖不足、redirect 参数契约错误，以及命令匹配器采用了与 Ikemen 完全不同的回看式算法。以“导入任意 MUGEN 角色”为目标，现状不能通过。

严重度：

- **P0**：会使合法角色逻辑静默变假、错误跳状态或无法出招。
- **P1**：常见角色存在行为差异，或 Oracle trace 无法收口。
- **P2**：Ikemen 扩展、多人模式、边界兼容和架构质量问题。

差异分类：

- **合理确定性设计**：为帧同步将 `float32` 换成定点、显式快照和哈希，不改变控制流语义。
- **兼容性专门设计**：有明确版本或配置开关，并能说明目标是 MUGEN 1.0/1.1 还是 Ikemen。
- **不规范简化**：缺少版本边界，静默返回 `0`/`Undefined`，或用近似算法代替目标引擎状态机。
- **结构性债务**：重复管线、职责错位、无法诊断或无法逐帧对照。

## 2. P0/P1 缺口总表

|级别|缺口|性质|直接后果|
|---|---|---|---|
|P0|未知标识符在 `ParseValue/ParseIdent` 中编译为常量 `0`|不规范简化|角色成功加载但对应控制器永久不触发，错误不可见|
|P0|未知函数保留参数值，不报错|不规范简化|例如 `foo(1)` 被当成 `1`，可能错误触发而非失效|
|P0|`Compile` 不检查剩余 token、括号和逗号的 `Expect` 不失败|不规范简化|部分表达式被截断后仍生成“有效”字节码|
|P0|`&&`、`||` 未由编译器生成短路跳转|语义错误|右侧 `random`、无效 redirect、除零等仍执行，RNG 序列与 Ikemen 分叉|

|P0|命令匹配采用“每帧回看整段历史”的贪心算法，而非 Ikemen 的逐步完成状态机|结构性语义错误|`time/steptime/>/同帧方向到按钮/重复方向` 均可能不同|

|P0|暂停期间直接跳过输入和命令更新|语义错误|Ikemen 默认 `command.buffer.hitpause=true`，当前无法在 hitpause 中缓冲招式|
|P1|`gametime`、`screenpos` 能编译但运行时返回 `Undefined`|实现断层|表达式看似支持，运行结果恒假/零|
|P1|`inguarddist` 固定简化为“对手 MoveType=A 且距离 <=160”|不规范简化|忽略 HitDef guard distance、多人目标、状态和缩放|
|P1|`animexist` 与 `selfanimexist` 共用当前 `AnimTable`|不规范简化|custom state、helper 借用动画时语义错误|
|P1|命令不支持 `steptime/autogreater/buffer.pauseend/buffer.shared`|兼容性缺口|双击、严格序列、暂停末尾输入和同名命令行为不同|
|P1|命令的 `>` 只检查相邻帧，不检查中间无关键变化|语义错误|Ikemen 会使步骤失效的杂键，当前仍可匹配|
|P1|不允许 Ikemen 的方向步骤与按钮步骤同帧完成|语义错误|`F,a` 等输入比目标引擎多要求一帧|
|P1|`$`、方向释放、对角释放只做位集合近似|不规范简化|冲突方向和释放边界与 MUGEN/Ikemen 不同|
|P1|存在两套表达式和两套命令系统|结构性债务|同一 CNS/CMD 经不同入口得到不同结果，测试可能验证错管线|

## 3. 表达式值与运算逐函数对照

### 3.1 `BytecodeValue`

对照 Ikemen：`bytecode.go` 的 `BytecodeValue`、`ToF/ToI/ToI64/ToB` 和工厂函数。

|当前函数|评估|
|---|---|
|`IsNone`、`IsUndefined`|显式 `Undefined` 标签代替 NaN，属于合理确定性设计。|
|`ToF`|`Undefined -> 0` 与 Ikemen 一致；但 `Int64` 先截成 `int` 再转定点，大整数与 Ikemen 不同。|
|`ToI/ToI64`|向零截断设计正确；`ToI` 的 int32 截断与目标接近。|
|`ToB`|`Undefined/0 -> false` 与 Ikemen 一致。|
|`None/Undefined/Float/Int/Int64/Bool`|工厂拆分清楚，可保留。|
|`ToString`|仅调试，不影响兼容。|

定点替代 `float32` 本身合理，但必须在 Oracle 中定义：整数、布尔和离散状态完全一致；位置/速度允许固定的 raw 或换算容差；三角函数和幂函数单独设误差预算。不能用“定点不同”掩盖控制流差异。

### 3.2 `BytecodeOps`

|函数|与 Ikemen 的差异与结论|
|---|---|
|`Neg/Not/BlNot`|基本一致。`Undefined` 经转换成为 0 后参与运算，这是 Ikemen 当前的兼容行为。|
|`Abs`|一般值一致；`int.MinValue` 在 C# `Math.Abs` 可能抛异常，应增加边界测试。|
|`Add/Sub/Mul/Div/Mod`|类型提升和整数除法方向基本一致；定点除法是合理差异。除零返回 `Undefined` 正确。|
|`Pow`|当前编译器把 `**` 做成右结合；Ikemen `expPow` 循环为左结合。`2**3**2` 会分别得到 512 与 64，是明确语义差异。旧 MUGEN 版本的整数幂 bug 分支也未实现，应通过版本策略决定是否需要。|
|`Eq/Ne/Gt/Ge/Lt/Le`|一般类型规则一致；定点精确相等与 float32 舍入不同，需要 trace 容差测试。|
|`And/Or/Xor/BlAnd/BlOr/BlXor`|单个运算函数基本一致；真正问题在编译器没有为 `&&/||` 生成短路控制流。|
|`Range`|先传播 `Undefined` 正确。当前“任一 Float”判断在现有类型集合下与 Ikemen 最小类型规则等价。|
|`Exp/Cos/Sin/Tan/Acos/Asin/Atan/Ln/Log`|确定性定点函数是合理改造；必须覆盖非法域、饱和/溢出和误差上界。|
|`Floor/Ceil`|仅 Float 转 Int、非 Float 原样返回，语义正确。|

相对 Ikemen 843f732，VM 运算函数还缺：`max/min/round/clamp/atan2/sign/rad/deg/lerp/randomrange` 等。前六项已进入 Ikemen trigger 语言；如果目标是纯 MUGEN，可标为扩展层，但应明确报“不支持”，不能降级为参数值。

## 4. 字节码执行器逐函数对照

### 4.1 `BytecodeExp.Run`

对照 Ikemen：`bytecode.go: BytecodeExp.run`。

正确部分：

- 栈式执行、8/32 位跳转、`OC_run/OC_nordrun`、redirect 失败压 `Undefined` 后跳过块，整体结构来自 Ikemen。
- 每条普通 opcode 后把当前上下文重置为原上下文，成功 redirect 和 `nordrun` 通过 `keepContext` 跳过重置，方向正确。
- 定点 literal 使用 8 字节 raw 是内部格式差异，只要编译器和 VM 同版本即可。

缺口与风险：

- 编译器从未为普通 `&&/||` 发出 `jz/jnz + pop`，所以执行器虽支持短路，实际 CNS 表达式不使用它。
- `OC_ifelse` 会先计算三个操作数，符合 Ikemen 的 `ifelse`，但不符合 `cond`；当前二者被编译成同一 opcode。
- 每次 `Run` 新建 `List<BytecodeValue>`，每个 `OC_run` 又 `Slice` 复制子字节码并新建 VM。高频 trigger 会产生持续 GC。帧同步上仍确定，但属于性能和架构债务。建议改为预验证的只读 code span + 每战斗实例栈池，不能使用跨战斗全局栈。
- `Pop/Peek` 在栈下溢时返回 `Undefined`，Ikemen 开发态直接 panic。产品版可以不崩，但编译阶段必须诊断；否则损坏字节码继续运行。
- `Peek32/ReadInt32/ReadInt64/Slice` 无边界校验。导入数据不应能直接制造字节码，但应给编译器产物做一次验证。

### 4.2 辅助函数

|函数|评估|
|---|---|
|`Binary/Unary/RangeOp`|简单清楚，语义由 `BytecodeOps` 决定。|
|`Pop/Peek/Swap`|容错策略需要从“静默 Undefined”改成可观测诊断。|
|`Jump8/Jump32`|偏移约定与 Ikemen 一致，需加边界和 `offset=0` 测试。|
|`Slice`|语义正确但有逐帧分配问题。|
|`ReadInt32/ReadInt64`|小端读取正确；需验证越界。|

## 5. 表达式编译器逐函数对照

### 5.1 `Compile/Tokenize/ParseFixed`

- `Compile` 没有检查 `End`，也没有返回错误对象。Ikemen `fullExpression` 明确拒绝剩余 token。**P0**。
- `Tokenize` 不支持科学计数、常量 `e/pi`、冒号及若干兼容 token；字符串不处理转义。更严重的是非法字符全部被当 `Op`，后续可能被吞掉。
- `ParseFixed` 截断到 9 位小数是合理的确定性策略，但需把舍入规则写入格式规范并用 Ikemen float32 输出做差分测试。
- 编译器实例持有 `_toks/_pos/_out` 可变状态，不可重入、不可并发。加载期单线程可以接受，但接口应标明，或将状态移入一次编译 session。

### 5.2 优先级函数

|函数|评估|
|---|---|
|`ParseBoolOr/ParseBoolAnd`|优先级正确，但只发 `OC_blor/OC_bland`，没有 Ikemen 的短路跳转。|
|`ParseBoolXor/ParseOr/ParseXor/ParseAnd`|基本对齐。|
|`ParseEqne/EmitRange`|范围语法基本对齐；`Expect` 不报错使残缺范围被接受。|
|`ParseGrls/ParseAddSub/ParseMulDiv`|一般优先级正确。|
|`ParsePow`|右结合，与 Ikemen 左结合不同。|
|`ParseUnary`|一般一元运算可用；redirect 内非法运算没有复刻 Ikemen 的限制。|

### 5.3 `ParseValue/ParseIdent/ParseFunction`

这是当前最高风险区域：

- `ParseValue` 无法识别时发 `0`。
- `ParseIdent` 未知标识符发 `0`。
- `ParseFunction` 未知函数不发 opcode，保留已压入的参数。
- `Expect` 缺少符号时什么也不做。

上述行为不是 MUGEN 容错的忠实实现，而是不可诊断的语言改写。正确边界应为：编译结果包含 `Success/Diagnostics/UnsupportedFeatures`；兼容模式可按角色版本决定是否降级，但每次降级都必须进入兼容报告和 trace 元数据。

逐项差异：

- `var/fvar/sysvar/sysfvar`：后两者错误复用前两者 opcode；`localvar/map` 缺失。
- `ifelse`：eager 语义基本对齐；`cond` 必须生成分支跳转。
- `const`：只支持 `MConstId` 中 38 个字段；未知字段返回 0。相对 843f732，缺少其余 data/size/velocity/movement 字段以及 stage/game 常量族。纯 MUGEN 模式至少要完整覆盖 Elecbyte 标准字段。
- `gethitvar`：当前只支持 21 个字段。相对 843f732，除此集合外的字段全部缺失，包括 velocity 各态、offset/add、chain/player/proj、damage/power/guard、fall damage/kill/time/envshake 等族。
- `animelem = n, op t` 的旧式第二比较未实现，源码注释已承认。
- 旧式 `projhit123/projguarded123/projcontact123` 未实现。
- 字符串使用 ASCII 单字节长度；Ikemen 使用 string pool，非 ASCII 角色名和命令名会损坏。

### 5.4 Redirect 编译函数

|函数|评估|
|---|---|
|`EmitRedirect`|`p2/root/parent` 的单值绑定结构基本正确；当前允许 redirect 内括号，Ikemen 为避免 MUGEN 崩溃会拒绝部分形式。可作为宽松扩展，但要有模式开关。|
|`EmitEnemyRedirect`|索引在原上下文求值、子表达式在目标上下文求值，结构正确；运行时只支持 1v1 index 0。|
|`EmitIndexedRedirect`|仅 partner，运行时仅 index 0。|
|`EmitTwoArgRedirect`|helper 的参数顺序与运行时一致；target 的运行时消费错误，是 P0。|
|`EmitP2TypeCompare/EmitRedirectedNameCompare`|封装方式可用；失败应保持 `Undefined`。|
|`AppendI32`|正确。|

缺失 redirect：`player`、`playerid`、`playerindex`、`helperindex`、`stateowner`。其中 `stateowner` 对 MUGEN custom state 是 P0；其余主要是 Ikemen/多人扩展 P2，但“任意角色”目标最终也应支持。

## 6. Trigger 运行时审计

### 6.1 `MChar.ReadTrigger`

已实现且总体可保留的核心：`time/stateno/prevstateno/statetype/movetype/ctrl/anim/pos/vel/facing/life/power/alive/random/command/name/id/palno/hitpause/hitcount/move*`、基础动画时间、实体计数、投射物接触时间、受击恢复、变量、部分 const/gethitvar。

明确错误或简化：

- `gametime` 和 `screenpos` 编译器会生成 opcode，但 `ReadTrigger` 无 case，返回 `Undefined`。
- `inguarddist` 是固定 160 距离近似，不能作为兼容实现。
- `numtarget` 不支持 id 参数语义。
- `numhelper/numproj/numexplod` 的 owner、id、player 过滤需要逐项对照 Ikemen；当前均通过单一 world 列表简化。
- `animexist/selfanimexist` 未按 `animPN` 与自身资源归属分离。
- `random` 使用共享确定性 RNG 是合理设计；但短路缺失会额外消费随机数，导致整个战斗从该帧开始分叉。
- 未实现 opcode 最终统一 `Undefined`，运行期本身可以接受；问题在编译器未报告“该 trigger 未实现”。

### 6.3 `ReadGetHitVar/JugglePoints`

- `JugglePoints` 的查询入口清楚，但目标选择和最大值规则需跟 Ikemen 的 target id、hit state 对照。
- `ReadGetHitVar` 未知字段返回 0，不可接受。应在编译期产生字段级诊断，运行期只接收已验证 id。
- 当前字段 id 是项目私有协议，可保留；不要依赖它与 Ikemen opcode 数值一致。

## 7. Redirect 运行时逐函数对照

### 7.1 `MChar.Redirect`

- `root/parent/p2`：1v1 基本可用；不存在时返回 null 并由 VM 产出 `Undefined`，正确。
- `enemy/enemynear`：只允许 index 0 并返回 P2。1v1 是有意简化，必须在能力报告中声明；多人模式不兼容。
- `target`：当前只弹一个值，和编译器压入两个参数不匹配。应对照 `targetTrigger(id,index)` 同时按 id 和索引选取。
- `helper`：按 root、helper type、match index 遍历，基本方向正确；排序必须固定为 Ikemen player/helper 创建顺序，并进入 trace。
- `partner`：只有单个直接引用，未实现 team member 索引和不存在规则。
- 缺少 `stateowner`，使 custom state trigger 上下文不完整。

### 7.2 `FindHelperRedirect`

- 过滤 destroyed、root 和 helper id 的逻辑合理。
- 使用 `ReferenceEquals` 判断 root 是本地对象模型设计，可接受，但回滚恢复后必须保证引用重链稳定。
- `helperType > 0` 才过滤意味着 id 0 与负值均表示任意；需逐项验证 Ikemen 对 `helper(0)`、`helper(-1)` 的规则。
- 列表顺序就是 match index 语义，必须用 Oracle 覆盖创建、销毁、同 id 多 helper。

## 8. Trigger 组与 CNS 编译

### 8.1 `MugenCnsParser.BuildTriggers/FlushController`

当前正确实现了：`triggerall` 全 AND、同号 `triggerN` 多行 AND、不同组 OR。

但与 Ikemen `compiler.go` 的差异为：

- Ikemen 强制每个控制器存在 `trigger1`；当前空 trigger 集被视为真。
- Ikemen 检查 `triggerN` 是否跳号；当前 `SortedDictionary` 会把 `trigger3` 压成第二个 OR 组。
- Ikemen 对常量 false/true 做明确折叠，但仍保留错误诊断；当前没有编译诊断。
- Ikemen 将整个 trigger 集编译成带短路跳转的一个表达式；当前逐表达式 `Run`，组级短路是正确的，但单个表达式内部不短路。
- 当前未知 controller 和未知 trigger 均可静默降级；这会让兼容报告低估真实缺口。

### 8.2 `MTriggerSet.Passes`

循环逻辑本身符合 AND/OR 结构，属于可保留实现。需要修改的是输入约束：只有 parser 已验证 trigger1、组连续性和每个表达式编译成功后，`Passes` 才可信。

## 9. 命令模型与输入缓冲逐函数对照

### 9.1 `MCommandKey/MCommandStep/MCommandDef`

数据结构覆盖了基础方向、按钮、`~ / $ > + |`，但缺少：

- `L/R/N` 和 `UL/UR/DL/DR`。
- `d/w/m` 按钮。
- 每命令 `StepTime/AutoGreater/BufferHitpause/BufferPauseEnd/BufferShared`。
- 逐步骤完成状态、步骤计时和 loop order。没有这些状态就无法实现 Ikemen `Command.Step`。

### 9.2 `MCommandBuffer`

`Push/Ago/Clear/Clone/WriteHash` 作为原始帧历史环形缓冲实现是确定的，可用于调试和 trace；但它不是 Ikemen 的 `InputBuffer`，不能单独承担命令语义。

- `_count` 可无限增长，长期运行会 int 溢出；应使用 long 或环形写指针 + 有效帧数。
- `WriteHash` 把全部槽位和累计 count 纳入哈希，确定但包含逻辑上已过期数据。只要 clone 一致不会错，但会增加无意义分叉面。

### 9.3 `MInputBuffer.Update/Step`

这是对 Ikemen signed key timers 的较好移植：按住为正递增，释放为负递减，保存前一帧，B/F 随朝向解析。定点无关，适合帧同步。

差异：

- 只支持 7 个按钮，缺 `d/w/m`。
- SOCD 固定成“相反方向都取消”。Ikemen 支持 allow/last/absolute/first/neutral 多模式。固定策略可以是产品配置，但不是兼容必然要求，必须进入对局配置和 trace hash。
- 当前 `MCommandMatcher` 完全不使用该 signed buffer，而是另用帧历史，导致最接近 Ikemen 的数据结构只服务硬编码基础动作。

## 10. 命令解析逐函数对照

### 10.1 `MCommandParser.Parse/ParseKey/ApplyKeyBody`

- `Parse` 对普通逗号步骤和 `+`/`|` 有基础支持。
- 同一步同时出现 `+` 与 `|` 时，Ikemen 报错；当前优先按 `|` 拆分并把剩余 `+` 留进 key body，最终静默产生 `None`。
- 空步骤、重复 modifier、无 key、未用 `+` 分开的多 key，Ikemen 都诊断；当前大多忽略。
- `>` 仅允许出现在整个 raw step 的第一个字符；Ikemen 在每个 key part 前都可读取并作用于整步。
- `~5B` 当前设置 `ChargeTime=5` 后把 `Release=false`。Ikemen 保持 `tilde=true`，并用上一帧 signed state 验证蓄力后释放。这是 P0。
- `/5B` 的数字 charge、`N/L/R`、扩展按钮和 remap 均缺失。
- 未知 key 变成 `IsButton=true, Bits=None`，应改成诊断，不应生成永不匹配的定义。

### 10.2 `MugenCmdParser.Parse`

- 只读取 `[Command]`，完全忽略 `[Remap]` 和 `[Defaults]`。
- `defaultTime/defaultBufferTime` 变量存在但从未被任何 section 更新，是“看似支持 defaults”的死代码。
- 缺少 `steptime/autogreater/buffer.hitpause/buffer.pauseend/buffer.shared` 的 section 默认值和单命令覆盖。
- `buffer.time` 未强制最小 1。
- 合法命令错误没有文件、section、命令名和行号诊断。

现有 12 个 CMD 中有 7 个 `[Defaults]`、7 个 `[Remap]`。多数 remap 是 identity，但 Terrarian 明确把默认 command time 设为 30；当前仍使用 15，已产生实际行为差异。

## 11. 命令匹配逐函数对照

### 11.1 `MCommandMatcher.Matches`

当前算法：要求末步骤在 age 0 成立，再在 `cmd.Time` 窗口内向过去贪心搜索其余步骤。

Ikemen 算法：每条命令保存 `completed[]/stepTimers/curtime/curbuftime/loopOrder`，每帧只用 signed `InputBuffer.State/StateCharge` 推进一步。

因此当前算法不是“为了帧同步的合理改写”，而是不同语义模型：

- 不支持 `steptime`。
- 无法实现 `AutoGreaterExpand`。
- 无法实现 `GreaterCheckFail` 对杂键边沿的失效处理。
- 默认要求不同步骤占不同历史帧；Ikemen 对方向到按钮序列允许同帧完成。
- charge 是历史区间搜索，不是当前/上一帧 signed charge 状态。
- 相同历史可能在多帧被重复重新识别，行为依赖最后一步是否仍被视为边沿。
- `time` 的起算点和过期边界与 Ikemen 的 `curtime` 不同。

结论：不建议继续给该贪心算法打补丁。应保留原始历史仅用于 trace，把真正 matcher 重构成 Ikemen `Command.Step` 的显式状态机。

### 11.2 `StepSatisfied/KeySatisfied/ResolveDir/DirHeld`

- `StepSatisfied` 的 AND/OR 基本直观。
- `KeySatisfied` 的普通按钮按下/释放边沿可用，但 charge、方向 release 和冲突方向不等价于 Ikemen `InputBuffer.State/StateCharge`。
- `ResolveDir` 的 B/F 朝向转换正确。
- `DirHeld($)` 简化为 `(frame & want)==want`。Ikemen 为兼容 MUGEN 使用 signed timer 的特殊 `$` 行为，尤其在冲突方向和对角释放时不同。

### 11.3 `MCommandList.Update/IsActive/Clone/WriteHash`

- 同名命令 OR 和 buffer 倒计时的表面行为存在。
- `Update` 先减 active、再回看匹配；没有 hitpause/pauseend/extratime 参数。
- `_active` 中归零项不移除，功能可用但状态逐渐积累。
- `Clone/WriteHash` 覆盖了现有状态，是合理帧同步实现。
- 缺少 Ikemen `Assert/ClearName/BufReset` 以及 `buffer.shared` 的同名命令互斥规则。

## 12. 战斗帧输入时序

`MBattleEngine.Tick` 在 `PauseBool` 时直接跳过 `CommandList.Update` 和 `MInputBuffer.Update`。Ikemen 的输入采样与命令推进会区分 hitpause、pause、施法者 movetime，并允许命令配置 `buffer.hitpause`、`buffer.pauseend`；其默认 `DefaultBufferHitpause=true`。

当前做法不是定点或帧同步要求，而是不规范简化。建议把流程拆成：

1. 每个模拟帧始终采样并推进原始输入 signed timers。
2. `Command.Step` 根据 `hpbuf/pausebuf/extratime` 决定步骤和 buffer timer 是否推进。
3. 状态机是否运行继续由 pause/movetime 控制。
4. trace 同时记录 raw input、resolved B/F、signed timers、每命令完成位和 buffer time。

## 13. 重复管线审计

工程仍存在两套表达式系统：

- `Game/Expr/ExpressionVM`：AST + `IEvalContext`，由旧 `Import/Cns/CnsParser` 和 ECS 系统使用。
- `Mugen/Expr/MugenExprCompiler + BytecodeExp`：当前 `MCharLoader/MBattleEngine` 使用。

也存在两套命令系统：

- `Game/Command/CommandMatcher + Import/Cmd/CmdParser + CommandSystem`：只支持非常简化的方向/按钮序列。
- `Mugen/Command/*`：当前角色战斗管线使用，但仍未对齐 Ikemen。

这不是合理的兼容分层，因为两套实现没有明确的版本、适用范围或共享测试契约。建议：

- 选 `Mugen/Expr` 和新的 Ikemen 式 command state machine 为唯一角色运行时语义核心。
- 旧 AST/Command 管线若只供早期 ECS 测试，应标记 obsolete 并逐步删除；若仍有产品用途，应改成唯一核心的 adapter，不能继续独立解释 CNS/CMD。
- 测试目录必须明确标注验证哪条运行管线，避免旧测试通过造成错误完成感。

## 14. 建议重构边界

不建议继续扩充 `MChar.ReadTrigger` 巨型 switch 或给回看 matcher 追加条件。建议的模块边界：

1. **Expression front-end**：lexer/parser/typed compiler，返回 bytecode、诊断、feature usage，不接触 `MChar`。
2. **Bytecode runtime**：无分配解释器、显式 evaluation frame、只接受已验证 bytecode。
3. **Trigger registry**：每个 trigger 有名称、参数 schema、版本、返回类型和 evaluator；编译器与兼容报告共用同一注册表。
4. **Redirect resolver**：基于稳定 player/entity id 和 state owner context，不把对象引用及参数栈约定散落在 `MChar`。
5. **Input sampler**：raw input、SOCD、朝向解析、signed timers。
6. **Command compiler**：完整读取 remap/defaults/command，输出不可变 command program 和诊断。
7. **Command runtime**：逐命令 `completed/stepTimers/curtime/curbuftime`，直接对照 Ikemen `Command.Step`。
8. **Compatibility profile**：MUGEN 1.0、MUGEN 1.1、Ikemen 的差异显式配置并进入战斗 hash。

## 15. 测试与 Trace 收口建议

### 15.1 表达式差分测试

为每个函数建立表驱动测试，并由同一输入生成 C# 与 Ikemen 输出：

- 类型组合：Int/Float/Bool/Undefined 的所有二元组合。
- 边界：0、-0、int min/max、定点最小分数、除零、非法 log/trig/pow。
- 结合性：`2**3**2`、连续比较、位运算与布尔混合。
- 短路：`0 && random`、`1 || random`、`0 && (1/0)`、失败 redirect 位于右侧。
- `ifelse` 与 `cond`：未选分支含 `random`、除零和无效 redirect。
- parser negative cases：未知 token、缺括号、缺逗号、多余 token、未知函数、非法 redirect。

trace 字段至少包含：expression id、bytecode hash、evaluation owner id、state owner id、RNG seed before/after、result type/raw、diagnostic id。

### 15.2 Trigger/redirect 差分测试

- 对第 6.2 节每个 MUGEN trigger 建一条存在性和返回类型测试。
- 对现有角色逐文件编译，断言 `UnknownTrigger=0`、`UnknownFunction=0`、`UnsupportedField=0`；不能再以“不抛异常”为通过标准。
- redirect 矩阵：root/parent/p2/enemy/enemynear/helper/target/partner/stateowner，覆盖不存在、id、index、多个同 id、销毁、custom state。
- `target(id,index)` 必须有专门栈平衡测试：执行后栈深只增加一个结果。
- custom state 中分别读取 `stateowner, name`、`root, var`、`selfanimexist/animexist`。

### 15.3 命令解析与匹配差分测试

- 直接移植 Ikemen `ReadCommandSymbols` 的合法/非法样例。
- 覆盖 `[Remap]`、`[Defaults]` 和所有单命令覆盖字段。
- 符号矩阵：U/D/B/F/L/R/N、全部对角、a/b/c/x/y/z/s/d/w/m、`~ / $ > + |` 及组合。
- 重点序列：QCF、DP、360、双击、同方向重复、蓄力释放、`F,a` 同帧、`/B+a`、严格 `>` 中插杂键。
- pause 矩阵：normal、hitpause、pause、superpause、movetime、pauseend。
- 朝向翻转：命令开始和完成之间换边。
- SOCD 每种策略都进入对局配置和测试。

命令 trace 每帧记录：raw bits、resolved U/D/L/R/B/F、各 signed timer、命令步骤完成位、step timer、curtime、curbuftime、completeframe、最终 active name 集合。

### 15.4 角色级门禁

对每个导入角色生成静态兼容清单：

- trigger/函数/redirect/const/gethitvar 使用集合。
- command symbols/default/remap 使用集合。
- 每项实现状态和目标 profile。
- 编译诊断必须为零，或存在明确 allowlist 与理由。

随后为每条命令由 command program 自动生成一组最短合法输入，分别喂给 Ikemen 和 C#，比较 command complete frame 与随后状态号。只有“同一帧命令完成、同一帧状态切换”才算动作状态机适配，而不是动画播放或 smoke 不崩溃。

## 16. 实施顺序

1. P0：编译诊断和 strict mode，禁止未知表达式静默变 0。
2. P0：修复短路、`cond`、pow 结合性、sysvar 和 target 参数契约。
3. P0：补 `stateowner`，修 trigger1/组连续性。
4. P0：以 Ikemen `InputBuffer.State/StateCharge + Command.Step` 重写命令 runtime。
5. P0：完整读取 remap/defaults 和 pause buffer 配置。
6. P1：按现有角色频次补 trigger，优先 `p2stateno/ailevel/numenemy/rootdist/stagevar`。
7. P1：补完整 MUGEN const/gethitvar/edge/team/round 语义。
8. P2：加入 Ikemen 扩展 profile、多人 redirect 和扩展按钮。
9. 删除或适配旧双重表达式/命令管线。
10. 用逐表达式、逐命令和逐状态 trace 将 Oracle 收口。

最终验收标准不应是“所有角色能载入”，而应是：导入期无静默降级；同一输入下，表达式结果、RNG 消耗、命令完成帧和状态切换帧与 Ikemen 843f732 在所选兼容 profile 内一致。
