  导入后所有招式都能出、离散行为对 Ikemen 对齐”的最终目标，整体进度约 50% 左右；按“普通 1v1 角色的根角色招式、
  helper、projectile、投技主干能跑”的招式执行目标，大概 65-70%。真正卡点已经从受击状态机/R-ENT 主干转移到
  Explod/numexplod、helper redirect/partner、BindTo*/TargetBind/ChangeAnim2、两段式伤害、juggle、AnimTime 即刻刷新、
  trigger 长尾、localcoord/palette/sound 表现与 R-ORACLE 差分。

  结构层面差异清单
  需要重点盯这几项，不是简单补几个 controller 能解决：

  1. StateBlock 树没有移植
     当前扁平化设计可读性好，但 Ikemen 的状态执行模型支持嵌套块、条件块、循环和块级 persistent。只跑 KFM 或简单 CNS 可以
     顶住，兼容复杂角色时会遇到结构性缺口。

  2. Char 运行态不是 Ikemen 完整 Char
     当前 MChar 已不只是核心子集，受击、倍率、roundstate、Pause、Tier B 字段、helper/projectile、custom state 已有。
     explod、target bind、camera/team/localcoord/表现态仍要进入运行态或表现事件，并纳入 Clone/Hash 或引擎快照。

  3. HitDef/GetHitVar 字段不完整
     当前命中已覆盖伤害公式、攻防倍率、numhits/hitonce、fall.damage/down.* 等主干，但 MUGEN 角色大量依赖 p2getp1state、
     spark/sound/envshake、cornerpush、完整 gethitvar()。这会直接卡投技、打击感和复杂连段。

  4. 实体系统缺失
     Ikemen 的 helper/projectile/explod 是大量角色的基础。当前 helper/projectile/custom state 主干已打通，owner/root/
     parent/target 基本可用；剩 explod 追踪、helper redirect、bind 类控制器和 partner/team 扩展。

  5. 回合/系统层缺失
     RoundState 主干已进来，但 lifebar、win pose、round intro 细节、camera、screenbound/stage 边界仍没系统化。

  6. 表现和逻辑事件还未分层完整
     Explod、PalFX、AfterImage、EnvShake、PlaySnd 等现在还没有统一事件/实体输出。后面要避免把表现层 Unity 对象直接塞进逻
     辑层。

  我的判断
  架构方向没有明显“推倒重来”的问题：模块拆分、纯 C# 定点、MChar 承载运行态、controller 类化、parser 接线都能继续推进。真
  正的结构风险是 状态块扁平化。如果目标是“兼容大量 MUGEN 角色”，以后大概率要补一个能表达 Ikemen StateBlock 嵌套语义的中
  间层；如果目标先是“KFM/常规角色能打”，当前扁平模型可以继续顶一段。

  下一步优先级已经变化：先做 Explod(numexplod) 最小追踪，再补 helper(id) redirect/partner 和 BindTo*/TargetBind/ChangeAnim2；
  随后做 R-DMG-PIPELINE 两段式、juggle、AnimTime 即刻刷新、trigger 长尾、localcoord/palette/sound 与 R-ORACLE。
  R-ENT 的硬骨头已经完成，后面主要是覆盖面、表现事件和差分验证。
