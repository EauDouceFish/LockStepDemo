
• 结论
  当前已经从“Ikemen/MUGEN 战斗内核的早期骨架”推进到“1v1 根角色战斗主干可验证”的阶段，但还不是可兼容任意 MUGEN
  角色的完整引擎。按“能导入真实角色并可靠战斗”的目标，整体进度约 30-35%；按“两个 KFM 能在 Unity 里站、走、跳、
  出招、发生命中、进入受击与回合结算”的最小 playable 目标，大概 55-60%。真正卡点已经从受击状态机转移到
  Pause 字段重构收口、Target/控制器接线、R-ENT 实体系统、表现事件与回滚确定性。

  最近已提交绿基线：7da5e26，dotnet test 444/444。
  当前主工作区不是绿的：存在未提交 Pause 字段重构，dotnet test 编译失败；旧代码仍引用 MChar.PauseTime/SuperPauseTime，
  但 Char.cs 正在改为 MPauseState Pause 聚合模型。先恢复编译绿是下一步第一优先级。

  结构层面差异清单
  需要重点盯这几项，不是简单补几个 controller 能解决：

  1. StateBlock 树没有移植
     当前扁平化设计可读性好，但 Ikemen 的状态执行模型支持嵌套块、条件块、循环和块级 persistent。只跑 KFM 或简单 CNS 可以
     顶住，兼容复杂角色时会遇到结构性缺口。

  2. Char 运行态不是 Ikemen 完整 Char
     当前 MChar 已不只是核心子集，受击、倍率、roundstate、Tier B 字段批已有一部分。但 helper、projectile、explod、
     target bind、custom state、camera/team/localcoord/表现态仍要进入运行态并纳入 Clone/Hash。当前 Pause 字段重构要先收口。

  3. HitDef/GetHitVar 字段不完整
     当前命中已覆盖伤害公式、攻防倍率、numhits/hitonce、fall.damage/down.* 等主干，但 MUGEN 角色大量依赖 p2getp1state、
     spark/sound/envshake、cornerpush、完整 gethitvar()。这会直接卡投技、打击感和复杂连段。

  4. 实体系统缺失
     Ikemen 的 helper/projectile/explod 是大量角色的基础。当前只有两个 root char。没有实体生命周期、owner/root/parent/
     target 体系，很多控制器无法真实实现。

  5. 回合/系统层缺失
     RoundState 主干已进来，但 lifebar、win pose、round intro 细节、camera、screenbound/stage 边界仍没系统化。

  6. 表现和逻辑事件还未分层完整
     Explod、PalFX、AfterImage、EnvShake、PlaySnd 等现在还没有统一事件/实体输出。后面要避免把表现层 Unity 对象直接塞进逻
     辑层。

  我的判断
  架构方向没有明显“推倒重来”的问题：模块拆分、纯 C# 定点、MChar 承载运行态、controller 类化、parser 接线都能继续推进。真
  正的结构风险是 状态块扁平化。如果目标是“兼容大量 MUGEN 角色”，以后大概率要补一个能表达 Ikemen StateBlock 嵌套语义的中
  间层；如果目标先是“KFM/常规角色能打”，当前扁平模型可以继续顶一段。

  下一步优先级已经变化：先把当前 Pause 字段重构收口，让主工作区恢复 dotnet test 全绿；然后合入 Codex 分支的
  Target* 与 Tier B 参数捕获；随后把 Pause/Width/ScreenBound/HitOverride/ReversalDef 等控制器接到 MChar 字段。
  这些完成后再开 R-ENT，因为 helper/projectile/explod/custom state 是“真实角色泛用兼容”的最大地基。
