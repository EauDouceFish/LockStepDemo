# 2026-06-07 续 16：MugenLive 同引擎口径 + 展馆招式入口执行面板

用户要求抛弃展馆“自写状态机”，按 `MugenLive_kfm` 的实现口径复刻 Ikemen/MUGEN 第一阶段，并先解决展馆不能搓招、倒地/状态机是否缺失的疑问。本轮核查结论：展馆没有独立状态机，当前与 `MugenLiveView` 一样使用同一套 `MBattleEngine` + `MStateMachine`；问题在于展馆只有键盘实时输入和轻拳/轻脚按钮，缺少可审计的 `Statedef -1` 招式入口面板，无法证明 QCF/升龙/前冲等复合命令已被消费。

模块 1（状态机/搓招链路 TDD）：
- 重写乱码测试文件 `CommandMoveTests` 为 ASCII 注释版本，保留单键攻击回归。
- 新增 KFM 真实复合命令端到端断言：`QCF_x -> state 1000`、`upper_x -> state 1100`、`FF -> state 100`。
- 这些用例穿过 `.cmd` 解析、`MCommandList`、`Statedef -1`、`ChangeState`、`MBattleEngine.Tick`，不直接改 `StateNo`。

模块 2（招式目录/展馆执行面板 TDD）：

- 新增 `CommandTransitionCatalogTests`，锁住 `command != "holddown"` 不得被误当成正向输入，并确认 KFM 的 QCF/upper/FF 入口存在。

- 展馆默认给 P1/P2 debug 满能量，便于超必/能量前置招式在展馆里被触发；真实战斗语义仍以引擎和测试矩阵为准。

剩余边界：
- “倒地后是否自动起立”仍取决于 common state/gethit/lie-down/selfstate 路径是否完整复刻 Ikemen；这不是展馆状态机问题，仍在 P0-D/P0-E 的受击、round context、trace 对齐范围内。
- 当前展馆提供任意 DEF 的可玩输入和招式入口执行面板，但不能声明“任意 MUGEN 角色全技能 Ikemen 等价完成”；总表中的 HitDef/guard/ReversalDef/HitOverride、localcoord、SND/ACT、Ikemen JSONL trace 等 P0 仍未收口。
