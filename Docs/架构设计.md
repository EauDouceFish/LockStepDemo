## 0. 定位与一句话方案

- **课题**：《动作游戏中网络同步方案研究：帧同步与状态同步》，对标**网易火影忍者手游**（定点数帧同步的商业产品）。
- **一句话方案**：

```
Ikemen GO 的【引擎结构】（Statedef + Controller + 表达式 VM，只抄设计、不 fork、不复制代码）
        +
MUGEN 资源【导入管线】（SFF→图、AIR→动画/判定框、CMD→搓招、CNS→角色数据）
        +
我们自己的【定点确定性】（FFloat + 逐帧 hash 对账 + 回滚框架）   ← 参考项目为上 Lua 而扔掉的
        =
比"照搬参考项目"更适合本课题的方案
```

- **关键取舍（已定，不再反复）**：
  1. **战斗模拟 0 Lua。** 角色行为 = **数据**（被定点表达式 VM 解释），不是 Lua 脚本。理由：Lua=double→跨机器不一致→杀确定性→杀回滚。Ikemen 自己也是这么分的（`bytecode.go`/`char.go` 纯 Go，Lua 只在 `script.go` 管菜单/外壳）。
  2. **保留定点数（FFloat）+ hash 对账。** 这是本课题最硬核的资产，也是火影手游级回滚的前提。
  3. **回滚先不实现、但框架留出。** v1 用现有延迟锁步；S13 再接 `RollbackPredictor`，战斗代码不动。

- **诚实边界**：MUGEN 角色"任意丢进去就能用"是理想态。21 天能到的是：**导入管线跑通 + 表达式 VM + ~20 个最常用 controller**，让**行为规整的角色（KFM 这类）能打**。复杂/怪角色（自定义 hack、版本 bug 依赖）随后"添砖加瓦"，且永远不可能 100%（连 Ikemen 都做不到）。

## 1. 三条铁律（编译期 / 评审强制，不可破）

1. **逻辑层纯定点、纯 C#**：`Assets/Logic/`（asmdef `Lockstep.Logic`，`noEngineReferences`）内**严禁** `float`/`double`、`UnityEngine.*`、`System.Random`、`Lua`、单例。一切数值走 `FFloat`，随机走 `FRandom(seed)`。
2. **一切跨子系统调用走接口**：`IInputProvider` / `ITransport` / `IPredictor` / `IExpressionVM` / `IAttackTable` 等。v1 可挂 Null/Static 实现，v2 换真实现，业务 System 不改。
3. **Component = 纯数据 + snapshot-safe**：每个 `IComponent` 必须实现 `Clone()`（引用字段深拷）与 `WriteHash()`（只混整数 raw）。**System / VM 无静态可变状态**——所有运行时态写回 Component，否则回滚后串状态。

> 这三条现有代码已经在守（`IComponent`、`World.Snapshot/ComputeHash`、无状态 System 单例）。MUGEN 化沿用，不破例。

## 3. 引擎模型：Statedef + Controller + Trigger（MUGEN/Ikemen 核心）

格斗引擎的灵魂：**引擎代码只有一份；角色 = 数据。** 加角色/加招式**永远不写新 C#**，只加数据。

- **StateDef（状态模板）** ≈ MUGEN `[Statedef N]`：声明姿势类型、物理、默认动画、入场参数 + 一组 controller。
- **StateController（带触发条件的动作单元）** ≈ MUGEN `[State N, x]`：`type=` 决定干什么（ChangeState/HitDef/VelSet…），`trigger=` 决定何时干。
- **Trigger / Expression** ≈ MUGEN 表达式：`time=0`、`movecontact=1`、`p2bodydist X < 50`、`command="a"` —— 每帧求值。**这就是为什么需要表达式 VM 而不是 Lua。**

### 每帧执行循环（StateMachineSystem 的核心）

```
对每个有 StateMachineC 的实体（按 Entity.Id 升序，保证确定性）：
  if Hitstop > 0: Hitstop--; continue           // 命中冻结
  // MUGEN 全局状态：先跑 -1(指令→改状态), 视需要 -2/-3
  RunState(stateNo = -1)                          // cmd 文件里的 State -1
  RunState(stateNo = entity.CurrentStateNo)       // 当前状态
  AdvanceTimeAndAnim(entity)                      // Time++, 动画推进

RunState(no):
  stateDef = character.States[no]
  for ctrl in stateDef.Controllers:              // 顺序敏感
    if VM.EvalBool(ctrl.Trigger, ctx):           // triggerall && (trigger1 || trigger2 …)
      Execute(ctrl, ctx)                          // 改状态/速度/动画/产生 HitDef…
      if ctrl 是 ChangeState 且已切走: break/重定向
```

### MUGEN → 我们的 schema 映射

| MUGEN | 我们的结构 | Ikemen 参考文件 |
|---|---|---|
| `[Statedef 200] type=S movetype=A physics=S anim=200 ctrl=0` | `StateDef{Id=200, StateType=Stand, MoveType=Attack, Physics=Stand, Anim=200, Ctrl=false}` | `compiler.go`(stateDef 解析) |
| `[State 200,1] type=HitDef trigger1=Time=0 …` | `StateController{Type=HitDef, Trigger=(Time==0), Params}` | `bytecode.go`(各 controller) |
| `trigger1 = AnimTime=0`、`p2bodydist X<50` | `IExpressionVM.EvalBool(...)` | `compiler.go`+`compiler_functions.go`(trigger 编译)、`bytecode.go`(trigger 求值) |
| AIR `[Begin Action N]` + `Clsn1/Clsn2` | `AnimData{Frames[]{Sprite,Off,Dur,Clsn1[],Clsn2[]}}` | `anim.go` |
| `.sff` 精灵图 | 导入器 → Unity Sprite + 调色板索引 | `image.go` |
| CMD `[Command] command=D,DF,F,a` | `CommandData{Motion,Button,TimeWindow}` | `command`/`input.go` |
| `var(x)/fvar(x)/sysvar(x)` | `VarsC`（int[] / FFloat[]） | `char.go` |
| 受击：P2 进 5000~ gethit 状态 + GetHitVars | `HitSystem` 写 `IncomingHitC` → 切受击状态（v1 简化） | `char.go`(gethit) |
| 回滚快照 | `World.Snapshot/Restore`（已有） | `rollback.go`/`state_clone.go` |

## 3b. 战斗模式：MUGEN-2D（v1）与 火影-3D（v2）双支路 + 隔离设计

**决策（2026-06-02）**：做两个**独立的战斗模式**，**v1 只做 MUGEN-2D**（纯 2D 平面对拳，完全仿 MUGEN）；**火影纵深版（3D）作为 v2 后续支路**，现在不碰。

**为什么必须分开、不做成一个可切换的 3 轴引擎**：
- 纵深改变**摇杆语义**：2D 里"上=跳、下=蹲"是 MUGEN 角色 CMD/状态写死的；3D 里"上下=纵深走位、跳跃另设独立键"。两套控制方案**互斥**。
- 因此纵深版**与 MUGEN 角色资源天生不匹配**（它们的指令、走位、受击全是 2D 平面假设）。纵深更适合"技能释放"型战斗，而非 MUGEN 的走位对拳。
- 强行合并 = 输入层和状态解释层到处 `if(mode)`，互相污染。分成两个模式更干净。

**两模式对照**：

| | **v1 MUGEN-2D（现在做）** | v2 火影-3D（以后） |
|---|---|---|
| 轴 | X 横向 + Y 高度（MUGEN 原生） | X 横向 + 纵深 + 高度 |
| 控制 | 上=跳 / 下=蹲（MUGEN CMD 原样） | 摇杆=地面走位(横+纵深)，跳跃独立键 |
| 碰撞 | 2 轴 AABB | 3 轴 + 纵深邻近带 |
| 角色资源 | 直接用 MUGEN 包 | MUGEN 招式数据 + arena 机制(自搭) |
| 渲染 | screenY=高度，sort 固定 | screenY=纵深+高度，sort 按纵深 |

**v1 坐标约定（MUGEN 原生，便于差分对齐 Ikemen）**：`Pos = (X 横向, Y 高度)`，Y=0 地面、**上为负**（MUGEN 惯例）。importer 对坐标是**恒等映射**（MUGEN X→X、MUGEN Y→Y，不换轴不翻符号），VM 的 `Pos Y`/`Vel Y` 与 Ikemen 完全一致 → 差分测试坐标可精确对齐。表现层负责翻成 Unity 屏幕坐标。
> 注：现有火影代码用 `(X,Y=纵深,Z=高度)`，会被新的数据驱动引擎取代；v1 新引擎采用上面的 MUGEN 原生 2D 约定。

**共享 vs 模式专属（隔离设计 —— "把关键逻辑隔离开"）**：
- **共享层（两模式同一份，约占 90%）**：全部角色数据（StateDef/Controller/AnimData/Clsn/HitDef/Command）、表达式 VM、状态机、命中数据。它们是"招式逻辑"，**与空间维度无关**。
- **模式专属（隔离在这 4 个 seam；v1 只实现 2D 版）**：
  1. `PhysicsSystem` —— 在哪些轴上移动 / 重力
  2. `CollisionSystem` —— 2 轴 AABB vs 3 轴 + 纵深带
  3. 输入→意图映射 —— 摇杆语义（在输入/命令解释层）
  4. View 投影 —— screenY + sortingOrder
- **实现纪律**：这 4 个 seam 用清晰边界（接口/策略点）隔开，命名带 `Mode2D` 标识；v2 = 新增 4 个并行实现 + 一套新控制方案，**绝不改共享层**。

## 4. 数据 Schema（草案，B 阶段定稿）

> 全部 `readonly` / 不可变静态数据，挂在 `World.GameData`（开局后只读，故不入快照）。运行时态一律进 Component。

```csharp
// 一个角色的全部静态数据（一次导入，多实体共享）
CharacterDef {
    int Id; string Name;
    Dictionary<int, StateDef>  States;     // 含负数状态 -1/-2/-3
    Dictionary<int, AnimData>  Anims;
    CommandData[]              Commands;
    CharConstants              Const;       // life/power/attack/defence/各种 velocity（全 FFloat raw）
    // 调色板信息(换色)由表现层用，不入逻辑哈希
}

StateDef {
    int Id;
    StateType StateType;   // Stand / Crouch / Air / LieDown
    MoveType  MoveType;    // Idle / Attack / BeingHit
    Physics   Physics;     // Stand / Crouch / Air / None
    int?  Anim;  bool? Ctrl;  FVector2? VelSet;  int? PowerAdd; int? Juggle;
    StateController[] Controllers;
}

StateController {
    ControllerType Type;          // 见 §10 子集
    IExpr Trigger;                // 编译后的触发表达式（triggerall + trigger1..n）
    // 类型相关参数：用编译后的 IExpr 持有（值参数也可能是表达式，如 value=ifelse(...)）
    Dictionary<string, IExpr> Params;   // 或针对热点 controller 用强类型子类
}

AnimData { int Id; AnimFrame[] Frames; int LoopStart; }
AnimFrame {
    int SpriteGroup, SpriteImage;   // → SFF 索引
    FFloat OffX, OffY;              // 绘制偏移（表现层用；逻辑只用 Clsn）
    int Duration;                  // tick 数；-1=永久
    FlipFlags Flip;
    ClsnBox[] Clsn1;               // 攻击框（攻方 active 时启用）
    ClsnBox[] Clsn2;               // 受击框
}
ClsnBox { FFloat X1, Y1, X2, Y2; }   // 相对角色原点；facing 镜像在运行时做

CommandData { string Name; InputSymbol[] Motion; ButtonMask Buttons; int TimeWindow; int BufferTime; }

HitDef {                            // controller 的强类型代表（命中是热点）
    HitAttr Attr;                   // StateType(S/C/A) + Class(N/S/H) + Type(A/T/P)
    int Damage, GuardDamage;
    int PauseTime, HitstopFrames;   // 打击感
    FVector2 GroundVelocity, AirVelocity;   // 击退（FFloat）
    int GroundHitTime, GuardHitTime;        // 硬直
    HitFlag HitFlag, GuardFlag;
    // sparkno/sound 等表现字段 → 交表现层，不入逻辑哈希
}
```

## 5. Component 改造清单

| Component | 状态 | 说明 |

## 6. System 执行顺序（每帧，顺序敏感）

```
1. CommandSystem    采输入入环形缓冲 + 搓招匹配 → CommandStateC
2. StateMachineSystem  跑 State -1(指令改状态) + 当前 State 的 controllers（VM 求值）
3. PhysicsSystem    按 Physics 类型施加 重力/摩擦/速度 → 位置积分 + 地面 clamp
4. AnimSystem       按 Duration 推进 AnimFrame，更新 AnimTime/ElemTime
5. CollisionSystem  攻方 active 的 Clsn1 vs 受方 Clsn2 → HitEvent（写候选）
6. HitSystem        结算 HitDef：伤害/硬直/击退/双方 hitstop → 受方切受击状态
```

确定性要求：所有遍历**按 Entity.Id 升序**；任何"先到先得"的判定用稳定顺序；新生实体 Id 单调递增（已有 `_nextId`）。

## 7. 确定性契约（课题命根子）

- 数值：全 `FFloat`。表达式 VM 内部运算全定点；导入期把 MUGEN 的小数（如 `vel=-4.0`、`accel=.35`）量化成 `FFloat.FromRaw/FromFraction`。
- 随机：只用 `World.Random`(`FRandom`，seed 入快照与哈希)。MUGEN `random`/`RandomRange` trigger 映射到它。
- 哈希覆盖：`Frame + Random.Seed + NextId + 全实体(按 Id 排序)`，与 `Snapshot` 范围一致（见 `World.cs` 注释）。新增可变态必须同步进 `Clone`+`WriteHash`+`Snapshot`。
- **热更新规则**：重载角色数据文件即可热更（行为是数据不是编译代码）。但**联机对战必须锁数据版本号**——两端版本不一致 = 必然分叉（恰好被 `DesyncDetector` 抓到）。热更仅用于开发/单机调试。

| 角色运行时 / 物理 / 受击 | `char.go`(14k) |
| AIR 动画 | `anim.go` |
| SFF 精灵图 | `image.go` |
| Clsn 碰撞矩形 | `rect.go` |
| 回滚 / 快照 | `rollback.go` `state_clone.go` `netplay.go` |
| 输入 / 指令匹配 | `input.go` |

**纪律重申**：Ikemen 是 Go + 浮点，**只读它的语义，用 C# + FFloat 重写**，不复制代码、不引入浮点。

## 12. 简化策略与"烂尾"风险登记（必读）

schema / VM / Component **不是 MUGEN/Ikemen 的 1:1 复制**，是**有纪律的简化**。原则：

> **简化"做多少"（少做 controller/trigger、简化命中路由），绝不简化"长什么样"（结构形状对齐 MUGEN 语义）。**

**A. 安全的简化（加数据/加 case 就能长大，绝不烂尾）**
- 先 ~20 controller / ~40 trigger，而非全量 → 加第 21 个 = 一个 enum case + handler，地基不动。
- 受击路由先简化（直接进受击状态，不走 MUGEN 5000~ GetHitVars 全套）—— **前提：HitDef 数据字段保持完整**。
- HitDef 表现字段(sparkno/sound)、特效、UI 先粗 / 转表现层占位。

**B. 危险的简化 = 形状捷径 = 死胡同（真正的烂尾源）。本设计已逐条堵死：**

| 禁止的形状捷径 | 后果 | 对策 |
|---|---|---|
| 硬编码 `if(time==0)` 代替真表达式求值 | 永远只能跑手写角色，导不了任意 CNS | **T0.4 起就是真 VM**：parse→AST/bytecode→定点求值；trigger 增量加，VM 架构从头就真 |
| trigger 砍成单 bool | 表达不了 `triggerall + trigger1..n` | 从头保留 OR-of-ANDs 结构 |
| 状态用 6 个枚举 | 装不下任意状态号(200/1000/5000) | **CurrentStateNo 用 int + 字典**，现在就换 |
| 砍掉 var/fvar/sysvar | 大量角色无声崩坏 | **VarsC 从 Phase 0 就在**，入 Clone/WriteHash |
| 动画只播帧、不记 AnimElem/AnimTime | `AnimElem=3` 类 trigger 全废 | **AnimC 记 ElemTime/AnimTime** |
| HitDef 数据字段也砍 | 命中信息丢失、后期补不回 | HitDef **数据形状完整**，只简化运行时处理 |

**C. 安全网**：用 **Ikemen 当 Oracle 差分测试**，任何简化若导致行为偏离，测试变红——**不会"以为做好了其实地基烂了"**。这是敢简化的底气。

**一句话**：会烂尾的是"形状捷径"，不是"功能少"。本设计把形状捷径全堵死（真 VM / int 状态号 / triggerall 结构 / var 系统 / anim 计时 / HitDef 数据完整），剩下的都是"加数据就能长"的安全简化。
