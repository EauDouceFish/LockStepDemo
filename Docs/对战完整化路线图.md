## 1. 重构评估结论（用户问：是否要照 Ikemen 那样重构现有代码？）

**结论：不做结构性重构，保持现有 C# 风格；只做「定点化保真修补」。** 依据：

| 维度 | Ikemen（Go） | 我们（C#） | 是否需对齐 |
|---|---|---|---|
| 控制器派发 | bytecode 编译 + `bytecode.go` 巨型 type-switch | 一控制器一 C# 类 + 虚 `Run(MChar)` | **否**。`移植方案 §2.8` 已授权此适配；行为等价（差分测试只校**可观测离散量**，bytecode 编码是内部细节）；C# 多态更可读可调试 |
| Char | 巨 struct | `MChar` 大类（镜像字段）| 否，已对齐 |
| 状态机 | 嵌套 `StateBlock` 树 | 扁平控制器列表 + persistent/ignorehitpause 行为移植 | 否，行为已移植（`MStateDef.cs` 头注已说明刻意不抄嵌套树）|
| 表达式 | bytecode VM | bytecode VM（已 1:1）| 已对齐 |

**为什么不重构**：把 5389 行改成 bytecode-controller 模型 = 巨大 churn、破全部测试、**零可观测收益**。
Ikemen 上 bytecode 是为「序列化 + 上千控制器性能」，我们不需要。

**但以下「保真修补」要随手做（不是重构，是改对）**——它们是真实的 Ikemen 逻辑偏离：

- ✅ **R-FIX-1 ChangeAnim 缺 `a==nil` 守卫**（2026-06-04 已修，commit 待填）：切到不存在动画曾冻死动画运行态 +
  表现层位置（KFM 跳起浮空根因）。已对齐 `char.go:4489 changeAnimEx`，加 `MChar.CanChangeAnimTo` 守卫
  到 ChangeAnim / `MStateDef.RunInit` / ChangeState·SelfState 内联 anim；`MugenLiveView.Render` 改为位置永不冻结。
- ⬜ **R-FIX-2 动画赋值集中化**：现 ChangeAnim / RunInit / ChangeState / `MAnimSystem.Play` 各自写 AnimNo。
  抽一个 `MChar.SetAnim(animNo)`（含存在性守卫 + prevAnimNo），所有处委托之，杜绝再漏守卫。低优先、零行为变化。
- ⬜ **R-FIX-3 借 Terrarian common1 → 换标准 common1**：现 KFM 借 `Terrarian/common1.cns`（它改过、
  State 50,3 切 anim44 无 `SelfAnimExist` 守卫）。应换**标准 MUGEN common1.cns**（带守卫）或转换 Ikemen
  `data/common1.cns.zss`。依赖 `R-TRG-SAE`（SelfAnimExist trigger）。

## 2. 任务分解（按「离可玩有多近」排序；同层可多 agent 并行）

### Tier A — 让两个角色真正能打（最高优先，离「可玩」最近）

#### ⬜ R-GHV — 受击状态机 5000–5160（**头号阻塞，现 ≈0% 可用**）
- 依赖：Tier-B 的若干受击控制器（见下）先到位；M7 命中（已有核心）。
- Ikemen 源：`char.go`（gethit 处理 / getHitVarSet）+ 标准 `common1.cns` Statedef 5000–5160。
- 内容：被击后整套数据态——5000 立受击 / 5010 蹲受击 / 5020 空中受击 / 5030 击退 /
  5040–5050 击飞·落地 / 5070 倒地 / 5080 起身 / 5100+ 空中被击 / 5150 KO。
  这些是**数据（common1）**，但依赖一批控制器与 trigger（HitVelSet/VelMul/DefenceMulSet/HitOver/HitShakeOver…）。
- 验收：构造「A 普攻命中 B」→ B 进 5000 → 走完 hitshake→slide→recover→回 0，
  **离散对照**各阶段状态号/动画号；`GetHitVar(*)` 全字段被正确读取。

#### ⬜ R-ROUND — 回合状态机（现 `StartRound` 只是授 ctrl 的 shim）
- 依赖：无（可与 R-GHV 并行）。
- Ikemen 源：`system.go` RoundState 流转 + `char.go` 5900/intro。
- 内容：intro → fight（授 ctrl）→ KO/time over → win pose → 下一回合 / 结算。回合数、胜负判定、计时。
- 验收：一局能走完 intro→fight→KO→结算；超时判血量高者胜（离散：roundstate 序列、winner id）。

#### ⬜ R-2P — 双人对战接通
- 依赖：R-GHV（要能互相打出反应才算「对战」）。
- 内容：`MugenLiveView` 现只画 1 人 → 渲染并驱动 Chars[0]/[1]；2P 输入（第二套键位）或先放木桩。
  引擎已支持 2 角色（`Tick` 双向 TryHit）。
- 验收：两个 KFM 站场上，P1 普攻命中 P2，P2 掉血+进受击（Unity 实测 + 一个 headless 双角色命中测试）。
