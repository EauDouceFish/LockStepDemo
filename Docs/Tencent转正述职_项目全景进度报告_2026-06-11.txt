一、执行摘要
================================================================================

1. 项目定位

本项目是一个基于 Unity 2022.3.55f1、C#、Fix64 定点数的 MUGEN 兼容格斗引擎研究项目。
核心目标不是简单制作一款格斗 Demo，而是验证并实现以下技术闭环：

（1）直接读取 MUGEN 角色资源：DEF、CNS、CMD、AIR、SFF、SND。
（2）复刻 IKEMEN-GO 的状态机、表达式、控制器、命中、实体和回合语义。
（3）将 IKEMEN 的浮点运行模型改造成定点确定性模拟。
（4）在确定性模拟之上实现帧同步、状态哈希、快照和最终回滚网络。
（5）通过 Unity 完成资源展示、1v1、选人、车轮战、AI 和联机客户端。

2. 当前结论

截至 2026-06-11，项目已经完成“可运行的 MUGEN 格斗引擎研究原型”，而不是只有架构或
静态解析器。核心战斗主循环、角色加载、命令识别、状态机、动画、碰撞、命中、受击、
Helper、Projectile、Explod、投技自定义状态、Pause/SuperPause、回合、1v1、简单车轮战、
简单 AI、延迟锁步和 KCP 输入中继均已有代码。

但项目目前不能对外宣称“完整复刻 IKEMEN”或“任意 MUGEN 角色导入即用”，也不能宣称
“回滚网络已完成”。最关键的缺口是：

（1）IKEMEN headless Oracle 尚未真正跑通，缺少同场景逐帧差分证据。
（2）网络仍是收齐输入才推进的 delay lockstep，未接入预测、回滚和重模拟。
（3）调色板、PalFX、AfterImage、阴影、镜像、音效、相机、完整舞台和 motif 等表现不完整。
（4）复杂 StateBlock、长尾 Trigger/Controller、完整 HitDef/GetHitVar 语义仍未全量覆盖。
（5）联机没有房间握手、选人同步、内容指纹、配置同步、断线恢复和服务端生产治理。
（6）当前有 50 个工作区变更项，其中 19 个未跟踪 C# 文件、约 2508 行代码尚未提交。

二、项目背景、目标与价值
================================================================================

1. 背景与挑战

格斗游戏网络对输入延迟、逐帧一致性、碰撞顺序和状态恢复极度敏感。MUGEN/IKEMEN 角色又
不是简单动画资源，而是一套包含表达式、状态定义、状态控制器、命令输入、受击语义和动态
实体的可编程内容系统。项目同时面对四类复杂度：

（1）兼容复杂度：需要理解并复刻 MUGEN/IKEMEN 的隐式语义和历史兼容行为。
（2）确定性复杂度：Unity 和普通浮点不能直接保证跨机器逐位一致。
（3）网络复杂度：格斗游戏需要延迟锁步或预测回滚，状态必须可快照、恢复和重模拟。
（4）内容复杂度：不同角色的 localcoord、palette、common1、控制器和脚本习惯差异很大。

2. 项目技术价值

项目已经形成以下可复用资产：

- 纯 C#、无 Unity 依赖的逻辑程序集。
- Fix64 定点数学和确定性随机源。
- MUGEN 表达式编译器与字节码虚拟机。
- 数据驱动状态机和控制器体系。
- MUGEN 资源读取链路。
- 可 Clone、可 Hash 的战斗运行态。
- 快照与恢复能力。
- 逐帧 Trace 和差分比较数据结构。
- Unity 表现层与逻辑层解耦。
- Loopback、KCP、服务器权威输入广播和 MUGEN 输入中继等多种网络实验链路。

3. 项目亮点

（1）没有为了快速出 Demo 而把战斗逻辑写死在 MonoBehaviour 中。
（2）逻辑层 asmdef 设置 noEngineReferences，绝大多数战斗代码可脱离 Unity 运行。
（3）角色行为由 CNS/CMD/AIR 数据驱动，方向与 IKEMEN 一致。
（4）定点数、确定性随机、Clone、WriteHash、Snapshot/Restore 为回滚预留了正确基础。
（5）测试规模已达到 751 项，覆盖表达式、解析、状态机、命中、实体、回合、团队战和网络。
（6）已经从单角色展示推进到 1v1、选人、3v3 turns、AI 和 KCP 联机原型。

8. 阶段八：网络与服务器

旧 ECS 网络链路：
- Bootstrap 支持 Loopback、HostAndPlay、ClientConnect。
- LockstepClient 以 30Hz 推进 World。
- 客户端发送 frame + InputLag 的输入。
- LockstepServer/Room 以 33ms tick 广播 FrameAuth。
- 缺帧时服务端沿用玩家 LastInput。
- NullPredictor 实际只在 FrameInputBuffer 有数据时推进，没有预测回滚。

新 MUGEN 网络链路：
- MugenLockstepSession 默认 InputLag=2。
- 每端采样本地 MInput。
- 输入按 frame/player 广播。
- 收齐同一帧所有玩家输入后才调用 Round.Tick 或 TeamMatch.Tick。
- 256 帧环形缓冲。
- 每帧可计算 Engine.ComputeHash。
- LoopbackNetBus 用于测试。
- TransportNetChannel 将 MInput 包装为 MugenInputMsg。
- MugenRelayServer 使用 KCP 收包并转发给其他连接。

准确结论：
新链路是 delay lockstep，不是 rollback。它在网络抖动或丢包时会停等；InputLag 只是固定输入延迟，
没有远端输入预测，也没有发现预测错误后的 Snapshot、Restore、Replay。

服务器现状：
- MugenRelayServer 只做输入广播，不跑战斗模拟。
- 没有正式房间、匹配、选人同步、内容校验和比赛状态。
- 没有鉴权、限流、防作弊、超时、重连、观战和日志审计。
- 当前联机要求两端手工使用相同 roster。
- host=P0、client=P1 由本地配置约定，不是服务器握手分配。

3. 每帧战斗流水线

MBattleEngine.Tick 的实际顺序：

（1）World.Events.BeginFrame。
（2）推进 Pause/SuperPause，并计算角色 PauseBool。
（3）推进 Projectile contact 时间。
（4）玩家输入写入 CommandList 和 InputBuffer。
（5）MActionSystem.Prepare 执行走、跳、蹲等硬编码基础动作。
（6）MStateMachine.RunFrame 执行负状态和当前状态。
（7）MPhysics.Step 和落地检测。
（8）MAnimSystem.Action 推进动画。
（9）DrainSpawns 创建 Helper 和 Projectile。
（10）StepProjectiles 和 StepExplods。
（11）玩家与 Helper 跨队稳定遍历并执行命中。
（12）FlushPendingDamage 统一应用本帧伤害。
（13）RemoveDestroyed 删除 Helper。
（14）推进帧号并供 ComputeHash 对账。

这个顺序是项目最重要的确定性契约之一。任何顺序调整都可能改变状态切换、动画帧、碰撞和伤害结果。

4. 客户端链路

传统客户端：
Bootstrap -> ITransport -> LockstepClient -> World -> BattleGameLogic -> View。

MUGEN 客户端：
MugenSelectScreen -> MugenMatchSetup -> MugenTeamBattleView -> MTeamMatch ->
MRoundSystem -> MBattleEngine -> MChar/View/HUD。

联机时：
MugenTeamBattleView -> MugenLockstepSession -> TransportNetChannel ->
KcpClientTransport -> MugenRelayServer -> 另一端。

5. 服务器链路

传统房间服：
KcpServerTransport -> LockstepServer -> Room -> FrameAuthMsg。

MUGEN 中继服：
KcpServerTransport -> MugenRelayServer -> 原样转发 MugenInputMsg。

两套链路目前没有统一。这是演进过程中的合理结果，但需要在下一阶段收口，否则会形成长期维护成本。

六、与 IKEMEN-GO 的差距
================================================================================

1. 架构差距

IKEMEN-GO 是完整游戏引擎，包含约 12 万行 Go 源码和大量 Lua 数据/界面脚本。项目当前产品 C#
约 2.7 万行，其中还包含旧 ECS、Unity View 和网络实验代码。规模不能直接代表质量，但说明当前
只是复刻了 IKEMEN 的核心战斗子集，而不是完整运行时。

最大结构差距：
- IKEMEN 的 Char/System 是高度完整的运行态，项目 MChar 仍是逐步扩展版本。
- IKEMEN 有 StateBlock 树和完整编译器，项目目前主要是扁平 StateController 列表。
- IKEMEN 的资源、渲染、舞台、相机、音频、菜单和模式统一在完整引擎中，项目目前分散在研究原型层。

2. 表达式和状态机差距

项目已覆盖常用表达式和 redirect，但仍缺：
- 完整 Trigger 集。
- 完整 Controller 集和逐字段语义。
- 嵌套 StateBlock。
- if/else、loop/for 等块结构。
- IKEMEN/ZSS 扩展。
- 版本兼容分支。
- 历史 bug compatibility。
- 所有 undefined、类型转换、整数/浮点截断和错误恢复细节。

风险：
简单 KFM 类角色可用不代表复杂角色可用。大量角色依赖边角 Trigger、Helper redirect、Target、
Explod、PalFX、HitDef 细节和自定义 common state。

3. 命中和角色语义差距

项目已有命中主干，但 IKEMEN 还包含更完整的：
- HitDef 字段。
- GetHitVar 字段。
- Guard、Reversal、HitOverride、Priority、Juggle。
- Corner push。
- Target ownership。
- 多实体接触顺序。
- Hit pause、Pause、SuperPause 交互。
- KO 同帧顺序。
- 版本和角色类型差异。

当前最缺的不是再增加几个字段，而是用 Oracle 在真实角色场景中逐帧证明顺序等价。

4. 动画与渲染差距

项目完成了逻辑动画推进和基础 SpriteRenderer，但 IKEMEN 还有：
- 完整 blend/trans。
- PalFX、AllPalFX、BGPalFX。
- AfterImage。
- remappal、ownpal。
- shadow、reflection。
- 插值、缩放、旋转和窗口裁剪。
- draw priority、layerno、sprpriority。
- camera 和 zoom。
- stage 背景层和视差。
- localcoord 的统一坐标转换。

项目目前很多控制器只产生确定性事件或保存参数，没有达到最终画面等价。

5. 音频差距

项目已有 SND 读取和 PlaySnd 类事件，但缺少：
- 真正的音频资源缓存。
- group/index 查找。
- channel 管理。
- lowpriority。
- loop、freqmul、volumescale。
- panning。
- Pause/SuperPause 下的音频行为。
- 舞台 BGM、角色语音和 motif 音效。

6. 舞台和相机差距

项目 MStage 当前主要是左右边界。IKEMEN 的舞台系统还包括：
- Stage DEF。
- 背景 SFF。
- 多层背景。
- delta、velocity、tile、mask、window。
- boundleft/right/high/low。
- camera startx/starty、tension、verticalfollow、zoom。
- player push 和 screenbound。
- shadow/reflection。
- 不同 localcoord 和分辨率。

7. 游戏模式和外壳差距

项目有 1v1 和简单 Turns。IKEMEN 还包括：
- Arcade。
- Versus。
- Team Simul。
- Team Turns。
- Tag。
- Survival。
- Training。
- Watch。
- Netplay。
- Replay。
- Continue。
- Storyboard。
- Select、Options、Lifebar、Motif 全套配置。
- Lua 外壳扩展。

8. 输入差距

项目主要是键盘位输入和简单 AI。IKEMEN 还支持：
- 完整键盘/手柄映射。
- 多控制器。
- Analog axes。
- SOCD resolution。
- Button assist。
- 输入配置 UI。
- AILevel 行为。
- Netplay 输入、Replay 输入和 Rollback 输入统一。

9. 网络差距

IKEMEN develop 当前源码明确包含 delay netplay 和 rollback netcode：
- rollback.go。
- SaveGameState / LoadGameState。
- AdvanceFrame 重模拟。
- LiveChecksum。
- Desync test。
- Replay 记录和回滚时间线截断。
- netplay 配置同步。
- 内容 fingerprint。
- host 配置覆盖和 peer ack。

项目目前：
- 有 MBattleEngine.Snapshot/Restore。
- 有 Engine.ComputeHash。
- 有 MugenLockstepSession。
- 但没有预测、回滚窗口、输入确认、错误预测检测和重模拟。
- MugenLockstepSession 的 hash 只计算 Engine，未组合 MRoundSystem.WriteHash。
- TeamMatch 联机 hash 也没有覆盖 ActiveIndex、BoutNo、MatchOver 等团队编排状态。
- Snapshot 当前主要覆盖 MBattleEngine，不覆盖完整 RoundSystem 和 TeamMatch 的可恢复状态。
- 没有把 hash 发到网络另一端比较。

因此 M10 仍然是明确未完成状态。

10. 测试和验证差距

项目优势：
- 751 个 .NET 测试。
- 测试可脱 Unity 快速运行。
- 已覆盖大量模块。

与“IKEMEN 等价”之间仍缺：
- IKEMEN headless 同输入 Trace。
- 同一角色、同一随机种子、同一输入脚本的逐帧对账。
- 离散字段全等。
- 定点和 IKEMEN 浮点连续量容差。
- 大规模角色冒烟矩阵。
- Unity PlayMode 自动测试。
- KCP 双进程端到端测试。
- 网络抖动、丢包、乱序和长时间 stall 测试。
- 1 小时以上 soak。
- GC、CPU、内存和实体峰值性能测试。


七、当前设计问题与风险
================================================================================

P0 风险 1：代码、提交和文档不同步

现象：
- HEAD 停在 2026-06-09 1v1 场景。
- 当前工作区有 50 个修改/未跟踪项。
- 19 个未跟踪 C# 文件约 2508 行。
- 团队战、AI、网络、调试、回合重构和新场景尚未形成可审计提交。
- 移植方案仍写 M10/M11 未开始，但代码已有 Snapshot、Unity View 和锁步会话。
- 部分文档仍写 491/509/687/709/727 测试，而当前实跑是 751。

影响：
- 无法明确判断稳定基线。
- 容易丢代码或在合并时产生大冲突。
- 述职时会出现文档数字互相矛盾。

建议：
- 立即按“回合重构、团队战、AI、网络、调试工具、文档”拆分提交。
- 创建唯一的 STATUS.txt 或 Generated/current_status.json，由脚本自动生成代码、测试和提交基线。

P0 风险 2：回滚能力没有闭环

现象：
- 有快照，但网络会话不使用快照。
- 有 Hash，但联机不交换 Hash。
- Round/Team 状态未纳入统一 Snapshot 和 Hash。
- 没有预测输入和错误预测修正。

建议：
- 建立 MatchStateSnapshot，统一包含 Engine、Round、TeamMatch 和网络时间线状态。
- 建立 Match.ComputeHash，禁止只用 Engine.ComputeHash。
- 实现 PredictionWindow、ConfirmedFrame、SaveFrame、RestoreFrame、Resimulate。
- 首先做离线 rollback sync test，再接 KCP。

P0 风险 3：没有 IKEMEN Oracle 证据

现象：
- 已经有 MBattleTraceRecorder 和 MBattleTraceComparer。
- 但没有完成 IKEMEN 侧 headless producer。
- 自动函数名匹配不能证明语义等价。

建议：
- 固定一个 IKEMEN commit。
- 在 IKEMEN 增加最小 trace 输出。
- 先选择 5 个黄金场景：站立、走跳、普通攻击命中、Projectile、投技。
- 离散字段逐帧全等；Pos/Vel 使用明确 Q20 或 Raw 容差。
- Oracle 测试进入 CI，任何控制器修改必须通过。

P0 风险 4：联机内容没有同步

现象：
- roster 由双方手工约定。
- 没有角色包 hash、common1 hash、配置 hash、seed 和版本协议。
- 中继服务器没有 room/match 概念。

建议：
- 增加 Hello、MatchConfig、RosterCommit、Ready、StartAtFrame、HashReport、Disconnect 消息。
- 对 DEF/CNS/CMD/AIR/SFF/SND/common1 和引擎版本生成内容指纹。
- 不一致时禁止开局。

P1 风险 5：两套引擎和两套网络并存

现象：
- Assets/Logic/Game 下旧 clean-room 引擎仍存在。
- Assets/Logic/Mugen 下新引擎已成为主线。
- LockstepClient/Room 和 MugenLockstepSession/Relay 各自实现不同模型。

建议：
- 明确旧链路只作为教学样例或直接归档。
- 将通用 Transport、Codec、连接管理保留。
- 将正式游戏统一到 MatchSession + MugenSimulation。
- 删除或移出生产 asmdef 的旧 PlayerStates、旧 ExpressionVM 和旧 Systems，避免误接。

P1 风险 6：网络会话健壮性不足

问题：
- 环形缓冲固定 256 帧，长 stall 可能覆盖旧帧。
- 重复包取最后值，但没有输入承诺，恶意端可修改已发送帧。
- 没有迟到帧、未来帧上限和窗口校验。
- 没有超时、重连、对局结束握手。
- KCP Update 在 View.Update 的模拟之后调用，可能增加一帧输入处理延迟。
- 中继服接受连接后全部互相广播，没有房间隔离。

建议：
- 增加 MaxFutureFrame、MaxRollbackFrames 和 LastCommittedFrame。
- 已提交输入不可更改。
- 网络 Update 在逻辑步之前。
- 使用 roomId/matchId 路由。
- 增加心跳、断线和 session token。

P1 风险 7：完整状态覆盖尚未审计

问题：
- 每增加 MChar 字段，都必须同步 Clone 和 WriteHash。
- Stage、NoDamage、Round、Team、FrameEvents 是否属于模拟态需要统一定义。
- 当前只有字段级人工纪律，缺少自动检查。

建议：
- 用反射测试检查 MChar 可变字段是否被 Clone/Hash 覆盖。
- 为 Snapshot -> Mutate -> Restore -> Hash 建通用属性测试。
- 为每个 Controller 增加两端相同输入的双实例哈希测试。

P1 风险 8：表现层与逻辑事件未形成正式协议

建议：
- 定义 PresentationFrame：
  EntityPose、SpriteState、VisualEvent、SoundEvent、CameraEvent、HudState。
- 逻辑只输出可重放事件。
- View 按 confirmed/predicted frame 区分一次性音效，避免回滚重复播放。
- Explod/Helper/Projectile 建对象池。

P2 风险 9：工程质量和 CI 不完整

建议：
- CI 至少执行 dotnet test、Unity batchmode compile、PlayMode smoke、格式检查、确定性扫描。
- 自动扫描 Assets/Logic 中 UnityEngine、System.Random、double、非允许 float。
- 自动生成函数审计、Controller/Trigger 覆盖和角色冒烟报告。
- 增加 Release build 和 headless relay build。


八、建议的优化方案
================================================================================

1. 第一优先级：建立可信基线

目标：让任何人能回答“现在到底完成到哪里”。

工作项：
- 清理并提交当前 50 个工作区项。
- 文档统一到 751/751。
- 生成 BuildInfo：Git SHA、IKEMEN SHA、Unity 版本、测试数、角色包指纹。
- 冻结 v0.1-prototype 标签。

验收：
- 干净工作区。
- 一条命令生成状态报告。
- 报告数字与 CI 一致。

2. 第二优先级：Oracle 差分

目标：把“看起来像 IKEMEN”变成“逐帧有证据”。

工作项：
- 完成 IKEMEN trace producer。
- 接入现有 MBattleTraceComparer。
- 建立 5 个基础场景、10 个真实角色、1000 帧黄金数据。
- 差异分为：离散错误、连续量漂移、表现差异、明确不支持。

验收：
- 基础场景离散字段 100% 全等。
- 连续量误差有明确阈值。
- 每个失败可以定位到字段和帧。

3. 第三优先级：真正回滚

目标：完成课题最核心的技术闭环。

设计建议：
- SimulationFrame：当前模拟帧。
- ConfirmedFrame：双方已确认的最大帧。
- InputHistory：每玩家真实/预测输入。
- SnapshotRing：每帧或每 N 帧快照。
- Predict：远端缺失时沿用方向或 None，策略可配置。
- Correct：真实输入到达且不同，恢复最早错误帧。
- Resimulate：无渲染快速重跑到当前帧。
- Presentation：只提交最终可见帧的一次性事件。

必须补的状态：
- Engine。
- RoundSystem。
- TeamMatch。
- RNG。
- Pause。
- Stage 可变态。
- 实体世界和生成队列。
- 输入时间线必要元数据。

验收：
- 120ms 延迟、20ms 抖动下不卡帧。
- 人工制造错误预测后最终 Hash 与无延迟基准一致。
- 10000 帧同步测试无分叉。

4. 第四优先级：内容和房间同步

目标：两端无需人工约定即可安全开局。

工作项：
- 房间创建与加入。
- P0/P1 分配。
- 选人同步。
- roster、stage、mode、seed、engine version、content fingerprint。
- Ready barrier。
- 开始帧同步。
- HashReport 和 desync dump。

- 先做复杂 UI 美化再修同步和状态覆盖。

6. 第六优先级：性能与可维护性

工作项：

- 热路径避免 Dictionary、List 临时分配和反射。

- 预编译角色数据，减少运行时 CNS 解析。
- 角色包加载缓存。
- Profiler 指标：Tick P50/P95/P99、GC alloc/frame、实体峰值、回滚重模拟耗时。

十、Tencent 转正述职口径总结
================================================================================

1. 目标承担

项目承担的是“格斗游戏确定性模拟 + MUGEN 内容兼容 + 网络同步”的复合课题。难点不是单个功能，
而是确保解析、状态机、命中、动画、实体、回合、客户端和网络在同一确定性时序下工作。

2. 核心结果

- 建立了纯 C# 定点逻辑层。
- 完成了 MUGEN 资源主干读取。
- 完成了 IKEMEN 风格表达式、状态机和大量控制器。
- 完成了玩家、Helper、Projectile、Explod 的统一战斗推进。
- 完成了 1v1、回合、3v3 turns、AI 和 Unity 可玩流程。
- 完成了 751 项自动化测试。
- 建立了 Snapshot、Restore、Hash 和 Trace 基础。
- 建立了 KCP delay lockstep 原型。

3. 技术判断与取舍

正确取舍：
- 战斗逻辑不引入 Lua/double。
- Unity 与逻辑层隔离。
- 角色行为数据驱动。
- 以 IKEMEN 源码作为 Oracle。
- 先做可测试逻辑，再接表现。

需要纠偏：
- 文档进度长期滞后于代码。
- 未提交工作区过大。
- 功能增加速度快于 Oracle 验证速度。
- “有 Snapshot”不能等同于“有 Rollback”。
- “能加载多个角色”不能等同于“兼容任意角色”。

4. 项目影响

该项目已经证明：
- MUGEN/IKEMEN 的核心战斗语义可以迁移到 C# 定点运行时。
- 逻辑层可以脱离 Unity 做高密度自动化测试。
- 角色包可以驱动 Unity 中的真实战斗，而非写死角色技能。
- 快照和哈希架构可以为后续回滚提供基础。

5. 后续目标

下一阶段的核心不应继续追求“模块数量”，而应追求三项可证明结果：

（1）Oracle：同一输入下与 IKEMEN 逐帧对齐。
（2）Rollback：高延迟和抖动下不中断，最终状态一致。
（3）Compatibility：10 至 20 个真实角色形成稳定、可重复的冒烟矩阵。

6. 最终评价

从实习/转正述职角度，本项目具备较强的技术深度、复杂系统拆解能力和工程产出密度。尤其是
定点确定性、状态机移植、资源解析、实体语义和测试体系，已经超过普通玩法 Demo 的范围。

当前主要短板不是缺少代码量，而是缺少“最终闭环证据”和“生产化治理”。只要完成 Oracle、
Rollback 和联机内容同步三个 P0 项目，项目就可以从“高完成度研究原型”升级为“可信的可演示
引擎原型”。若再补齐表现、舞台、音频和模式系统，才有资格进一步讨论 IKEMEN 级完整引擎。
