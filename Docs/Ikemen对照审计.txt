0. 方法论与符号
--------------------------------------------------------------------------------
- 对照粒度：每个我方文件映射到 Ikemen 源文件；逐关键函数列「我方 / Ikemen源(行) / 结构差异 / 效果是否一致 / 风险」。
- 效果一致性分级：
    [一致]   离散可观测量（状态号/动画号/命中/帧序/KO）在支持范围内逐位等价；连续量容差内。
    [近似]   常见路径等价，特定输入/边界分歧（注明触发条件）。
    [偏差]   行为不同或缺失，会改变可观测结果（注明潜在风险与影响面）。
    [装配]   我方为 headless 装配/适配层，Ikemen 对应代码耦合渲染/网络/SDL，非逐行可移植（注明这是设计决定还是简化）。
- 风险等级：高=影响真实角色对战正确性；中=特定招式/角色分歧；低=罕见/边缘/纯表现。
- 全局设计前提（贯穿全文，不再重复）：
    a) 浮点→定点：Ikemen 用 float32，我方用 FFloat(Q31.32)。连续量必然有定点截断差，离散量要求等价。
    b) bytecode 编译模型差异：Ikemen 把 CNS/表达式编译成字节码（compiler.go），运行 bytecode.go 巨型 switch；
       我方表达式同样编译字节码（Expr/），但「控制器」用一控制器一 C# 类 + 虚 Run()，非 bytecode 控制器。
       这是 CLAUDE/移植方案 §2.8 授权的适配，差分只校可观测离散量。本审计据此把「控制器派发模型差异」视为[装配]而非[偏差]。
    c) 单角色实体模型：我方 v1 无 helper/projectile/explod 实体系统（R-ENT 未做）。凡依赖多实体的函数标[偏差·缺失]。
    d) 主循环：Ikemen system.go fight()/action() 约 3000 行耦合渲染/血条/网络；我方 MBattleEngine.Tick 是重写的
       headless 装配。子系统（状态机/命中/动画/物理/表达式）逐行移植带源码行号；顶层编排为装配适配。

本审计独立评估「子系统移植质量」与「装配层等价性」两件事，不混为一谈。

模块 1：Expr 表达式 VM   ↔   bytecode.go（值/运算/执行器）+ compiler.go（编译）
================================================================================

[1.1] BytecodeValue.cs  ↔  bytecode.go:1074-1145 (BytecodeValue/ValueType/ToF/ToI/ToB)
  结构差异：
    - Ikemen：{vtype, float64 value} —— int/bool/float 全塞一个 float64；undefined = vtype 或 NaN。
    - 我方：{vtype, long _ival, FFloat _fval} —— int/bool 存 long、float 存 FFloat；undefined 仅靠 vtype（定点无 NaN）。
    - ValueType 枚举顺序 None=0/Float=1/Int=2/Bool=3/Undefined=4 —— 与 Ikemen 逐位一致（关键：Min(vtype) 规则依赖此序）。
  效果：[一致]
    - ToI：Ikemen int32(float64) 向零截断；我方 float 路径 TruncToLong 也向零截断、int 路径直取 —— 等价。
    - ToF/ToB：undefined→0/false，定义等价。
  风险：低。
    - Ikemen int 存 float64：int32 在 float64 尾数(52bit)内精确，无损；我方 long 同样精确 —— int32 域内两者皆精确。
    - 定点 vs float32 极值域不同（FFloat≈±2^31、float32≈±3e38）：超大数值溢出行为分歧，游戏量级用不到。

[1.2] BytecodeOps.cs  ↔  bytecode.go:1370-1530 (add/sub/mul/div/mod/pow/cmp/range)
  结构差异：Ikemen 用指针原地改 v1；我方不可变值返回新 BytecodeValue。语义等价。
  效果：[一致]（除一处）
    - 类型规则 Min(v1.vtype,v2.vtype)==Float 走浮点否则整数 —— 与 bytecode.go:1451 等逐行一致。
    - 比较用 .Raw 整型比（定点位级）对齐 float 比较的语义。
    - 除零：div 两边都 → Undefined（bytecode.go div 实测 *v1=BytecodeUndefined()）—— 一致。
    - pow：任一 float 或指数<0 → 浮点幂，否则整数快速幂 —— 与 bytecode.go:1370 一致。
  [偏差·低] 模零(mod by 0)：Ikemen 保留被除数 v1 不变 + 打印错误；我方返 Undefined。
       风险低：真实 CNS 极少 mod 0；分歧仅在该错误表达式的后续运算，且都属"脏输入"。

[1.3] BytecodeExp.cs  ↔  bytecode.go run()/evalF/evalI 执行器
  结构：栈机，逐 OpCode 解码执行；短路 jmp/jz/jnz、redirect run/nordrun、trigger 钩子 ReadTrigger/Redirect。
  效果：[一致]（表达式求值路径）；trigger/redirect 的"被读量是否齐全"取决于 Char.ReadTrigger 覆盖度（见模块 2）。
  风险：见模块 2 的 trigger 覆盖清单——未覆盖的 trigger 走 default→Undefined，可能误判门控（已知风险，逐个补全中）。

[1.4] MugenExprCompiler.cs (821行)  ↔  compiler.go + compiler_functions.go (约 1.5万行)
  结构差异：[装配/简化]
    - Ikemen：完整词法+递归下降，编译所有 trigger/函数/redirect 为 bytecode；带类型推导、常量折叠、错误定位。
    - 我方：等价的递归下降优先级链（expBoolOr..expValue 照搬），但 trigger/函数表是"按需登记的子集"。
  效果：[近似——覆盖度驱动]
    - 已覆盖的运算符/优先级/区间[a,b]/redirect(p2/root/parent/enemy)/statetype 字母枚举/command/animelem/距离/name
      等与 Ikemen 语义一致。
    - 未登记的 trigger/函数 → 压 0 或保留参数值（容错降级），与 Ikemen "编译期报错"不同：我方不报错而是静默降级。
  [偏差·中] 未覆盖 trigger 静默返 0：真实角色若用到未登记 trigger，门控可能误判（招式早出/不出）。
       影响面：随覆盖度提升收敛；本会话已补 random/animelem/距离/name/roundstate/inguarddist/animelemtime。
       风险：中——需要一份"已覆盖 trigger/函数 vs Ikemen 全集"的差集清单持续追踪（见模块 2 附表）。
  [偏差·低] 错误处理哲学不同：Ikemen 编译期严格报错；我方运行期宽容降级。利于吃脏数据，弊在掩盖角色脚本笔误。

[2.3] ReadTrigger 触发器覆盖（用量加权，基于本会话扫 12 角色真实 .cns/.cmd）

  [偏差·缺失·中] 依赖 R-ENT 实体系统的 trigger 全缺，走 default→Undefined/0：
    ishelper(862)/numhelper(403)/helper(241,redirect)/partner(124)/numenemy(70)/numexplod/numproj/numprojid/

  [偏差·缺失·低] 杂项 trigger 未覆盖：ailevel(203,返0=无AI，正确)/stagevar(58)/teammode/teamside/

[2.4] Clone/WriteHash 确定性  ↔  state_clone.go（Ikemen 用代码生成的 clone）
  结构差异：[装配] Ikemen 有 state_clone.go 自动生成的深拷；我方手写 Clone/WriteHash。
  效果：[一致·自洽] 我方契约=每个运行态字段进 Clone(深拷)+WriteHash(混入)；redirect 链接浅拷待引擎重链。
  风险：
    [偏差·中] 手写 Clone/Hash 的"漏一个字段"风险：新增字段若忘记加入 Clone/Hash 会破回滚一致性。
       缓解：本会话所有新字段都配了 Clone/Hash 往返单测（TierBFieldsCloneHashTests 等）；
       但缺少"全字段反射校验"的自动守卫——建议补一个反射断言所有 public 字段都被 Hash 覆盖的测试（R-DET 范畴）。
    [偏差·中] 共享可变态（Rng）只在引擎层 Hash 一次、Clone 浅拷引用：当前无回滚，正确；
       接 R-ROLLBACK 时必须由引擎级快照统一重链 Rng + redirect 图，否则克隆体共享旧 Rng → 串状态。已在代码注释登记。

[3.1] MStateMachine.RunFrame  ↔  char.go:11610 actionRun()
  Ikemen 真实顺序：state-4(内部,即便暂停也跑) → if !pauseBool{ state-3 → state-2 → state-1 → stateChange2(应用缓冲切换)
    → 当前态 } → 守招指令。-3/-2/-1 各带门控：-3 需 playerNo 自有且(player或keyctrl[2])；-2 需 player或keyctrl[1]；
    -1 需 playerNo 自有且 keyctrl[0]。
  我方顺序：(PendingStateNo>=0→ApplyTransition) → RunNegativeState(-3,-2,-1) → ApplyTransition → RunCurrentState → Time++。
  效果：[一致·针对玩家角色]
    - 负状态执行顺序 -3→-2→-1→当前 与 Ikemen 一致；缓冲切换在 -1 后当前态前应用，一致。
  [偏差·低] state -4 未实现：Ikemen -4 是引擎内部状态，标准 MUGEN 角色不定义 -4 → 对 MUGEN 数据无影响。
  [偏差·中(helper)] -3/-2/-1 的 keyctrl/playerType 门控未复刻：我方对存在的负状态无条件跑。
    玩家直控角色（keyctrl 全开）等价；helper（keyctrl 部分关）会多跑负状态 → 行为分歧。属 R-ENT 范畴。
  [偏差·中] 暂停粒度：Ikemen pauseBool 只挡 -3/-2/-1/当前态（-4 仍跑、输入缓冲另算）；
    我方在引擎层 frozen 跳过整帧（含输入缓冲、物理、动画）。详见模块 9 暂停审计——这是已知装配层近似。

[3.2] persistent / ctrlsps  ↔  char.go:6500-6510 + bytecode.go StateBlock.Run
  Ikemen：每控制器一个 ctrlsps 计数；persistent>1→执行后设大数锁定/倒计时；进状态时重建 ctrlsps 全零。
  我方：PersistGate/ResetPersist 复合键(状态号*256+序号)倒计时模型；进状态 ClearStatePersist 清目标区间。
  效果：[近似]——当前态的 persistent 帧冷却语义对齐。
  [偏差·低] 负状态 ResetPersist 时序与当前态不完全一致（代码注释/memory 已登记的边界）：
    负状态计数器从不被"进入"故不清零，长期运行的相位与 Ikemen 在 persistent=N 的精确节奏上可能差 1 帧。
    风险低——persistent=N(非1)在负状态较少见；但属真实分歧，需差分 trace 校验（R-ORACLE）。

[3.3] ChangeState 同帧重入 + hitpause  ↔  char.go stateChange1/stateChange2 + changeStateNest
  效果：[一致] hitstop>0 冻结当前态控制器（仅 ignorehitpause 跑）、ChangeState 同帧重入上限保护、
    statedef 头部 RunInit 求值（type/movetype/physics/anim/ctrl/velset 表达式）—— 与 Ikemen 行为对齐。
  [偏差·低] changeStateNest 嵌套深度上限：我方用固定 MaxReentry 常量；Ikemen 有自己的嵌套计数。极深递归边界分歧，罕见。

[3.4] MStateDef.RunInit（statedef 头部）  ↔  bytecode.go stateDef.Run + char.go StateBytecode.init
  效果：[一致] type/movetype/physics 字面量 + anim/ctrl/velset/facep2/juggle/poweradd 表达式求值，
    movetype 默认 I(bytecode.go:15139)。
  [偏差·低] 头部 persist 标志（hitcountpersist/movehitpersist/juggle 重置等"进状态清连段计数"）部分未做。
    风险低-中：影响连段计数在跨状态的保留，特定连段 combo 计数分歧。已登记。

[9.4] ★ Pause/SuperPause 暂停模型  [偏差·中·本审计重点]
  —— 这是被指出"随意"的部分，下面如实列差异。
  Ikemen 真实模型（system.go + char.go）：
    - sys.pausetime / sys.supertime 是【全局】系统状态(system.go:49/55)，主循环开头递减(system.go:2562)。
    - 每角色另有 c.pauseMovetime / c.superMovetime(char.go:3128) = 暂停期该角色还能动多久。
    - actionPrepare(char.go:11421)算 pauseBool：supertime>0 时 = (superMovetime==0)；pausetime>0&&pauseMovetime==0 时 = true。
    - c.acttmp(1 unpaused/0 default/-1 hitpause/-2 pause)贯穿全引擎门控：物理 canmove=acttmp>0(char.go:11944)、动画、镜头。
    - 受击方 hitshake 期间被强制提升 movetime(char.go:11036 getter.superMovetime=Max(..,hitshaketime))。
    - 专设 prevSuperMovetime/prevPauseMovetime 修 PauseTime 触发器 off-by-one(char.go:11431)。
  我方现实现（MBattleEngine.Tick）：
    - 暂停时长放在【per-char】SuperPauseTime/PauseTime，用 max 聚合成全局 → 与全局 sys.supertime 模型【结构不同】。
    - 用临时 frozen[] 数组在引擎层跳过被冻结角色【整帧】(含输入/状态机/物理/动画)；无 acttmp 贯穿门控。
    - 计时在 Tick 末尾递减；【无】prevMovetime off-by-one 修正；【无】受击 hitshake 提升 movetime。
  效果一致性：[近似——仅 1v1 单施放者]
    - 单角色发动 super、对方冻结、自己 movetime 窗口可动、计时到点恢复 → 离散结果与 Ikemen 大体一致。
  潜在风险（不一致点）：
    [中] 多施放者/受击叠加：Ikemen 全局 supertime + 各自 movetime + getter 提升那套，我方 max 聚合无法表达
         "受击方在被超必杀打中时按 hitshaketime 提升 movetime" → 受击方冻结时长可能与 Ikemen 差，连段/受身时序分歧。
    [中] PauseTime/SuperPauseTime 触发器：我方未暴露这两个 trigger，且无 off-by-one 修正 → 若角色用 pausetime 触发器
         判断暂停剩余帧，结果会错。
    [低] 暂停粒度：我方冻结整帧(连输入缓冲一起停)，Ikemen 暂停期仍处理 state-4/部分输入 → 边缘分歧。
    [低] 暂停期命中：我方 anyPause 时整体跳过 TryHit；Ikemen globalCollision 也在暂停外，但 spr/帧时序细节不同。
  结论：暂停是【已知的装配层近似】，不是 1:1 移植。当前 1v1 单施放可用；要对账 Ikemen 必须重构为
    "引擎全局 super/pause time + 每角色 movetime + acttmp 门控 + getter 提升 + off-by-one 修正"。已列入待办。

"最终能达到的效果是否一致"的总评

- 离散主干（命中→受击态 5000 系→恢复→扣血→KO、搓招→招式态、走跳蹲、伤害公式、随机、距离/名字/动画触发器）：

- 不一致集中在三类：①多实体(R-ENT) ②暂停/受击时序的装配近似(R2/R3/R7) ③未覆盖长尾(R6)+未驱动子系统(R4/R5)。
- 连续量(位置/速度)为定点容差一致，需 R-ORACLE 真 trace 量化漂移。

到达"与 Ikemen 离散全等"的必要工作（按依赖）：
  R-ENT 实体系统 → 伤害两段式+KO 排序(R3) → Pause 重构为全局态+movetime+acttmp(R2) → juggle(R5) →
  AnimTime 即刻刷新(R7) → trigger/控制器覆盖补全(R6) → R-ORACLE 差分 trace 量化连续量 + 兜底离散回归。
